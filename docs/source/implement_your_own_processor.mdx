# 实现自定义机器人处理器（Robot Processor）

在本教程中，你将学习如何实现自己的机器人处理器。
首先探讨为什么需要自定义处理器，然后以 `NormalizerProcessorStep` 为示例，解释如何实现、配置和序列化处理器。最后列出 LeRobot 附带的所有辅助处理器。

## 为什么需要自定义处理器？

在大多数情况下，当从传感器读取原始数据或模型输出动作时，你需要处理这些数据以使其与目标系统兼容。例如，一个常见的需求是对数据范围进行归一化，使其适用于神经网络。

LeRobot 的 `NormalizerProcessorStep` 处理了这一关键任务：

```python
# 输入：原始关节位置，范围 [0, 180] 度
raw_action = torch.tensor([90.0, 45.0, 135.0])

# 处理后：归一化到 [-1, 1] 范围，用于模型训练
normalizer = NormalizerProcessorStep(features=features, norm_map=norm_map, stats=dataset_stats)
normalized_result = normalizer(transition)
# ...
```

其他常见的处理需求包括：

- **设备放置**：在 CPU/GPU 之间移动张量并转换数据类型
- **格式转换**：在不同数据结构之间进行转换
- **批处理**：为适应模型而添加/移除批处理维度
- **安全约束**：对机器人命令施加限制

```python
# 组合多个处理器的示例管道（pipeline）
pipeline = PolicyProcessorPipeline([
    RenameObservationsProcessorStep(rename_map={}),
    AddBatchDimensionProcessorStep(),
    NormalizerProcessorStep(features=features, stats=stats),
    DeviceProcessorStep(device="cuda"),
    # ...
])
```

LeRobot 提供了一套管道机制来实现输入数据和输出动作的处理步骤序列，使你可以轻松地以正确的顺序组合这些转换，以获得最佳性能。

## 如何实现自己的处理器？

我们将以 `NormalizerProcessorStep` 为主要示例，因为它展示了处理器的核心模式，包括状态管理、配置序列化和张量处理，这些是你在日常开发中经常需要的。

准备解决问题所需的处理步骤序列。一个处理器步骤（processor step）是一个实现了以下方法的类：

- `__call__`：对输入转换（transition）执行处理步骤。
- `get_config`：获取处理器步骤的配置。
- `state_dict`：获取处理器步骤的状态。
- `load_state_dict`：加载处理器步骤的状态。
- `reset`：重置处理器步骤的状态。
- `feature_contract`：显示处理器步骤对特征空间的修改。

### 实现 `__call__` 方法

`__call__` 方法是处理器步骤的核心。它接收一个 `EnvTransition` 并返回一个修改后的 `EnvTransition`。以下是 `NormalizerProcessorStep` 的工作方式：

```python
@dataclass
@ProcessorStepRegistry.register("normalizer_processor")
class NormalizerProcessorStep(ProcessorStep):
    """使用数据集统计信息对观测/动作进行归一化。"""

    features: dict[str, PolicyFeature]
    norm_map: dict[FeatureType, NormalizationMode]
    stats: dict[str, dict[str, Any]] | None = None
    eps: float = 1e-8
    _tensor_stats: dict = field(default_factory=dict, init=False, repr=False)

    def __post_init__(self):
        """将统计信息转换为张量以提高计算效率。"""
        self.stats = self.stats or {}
        self._tensor_stats = to_tensor(self.stats, device=self.device, dtype=torch.float32)

    def __call__(self, transition: EnvTransition) -> EnvTransition:
        new_transition = transition.copy()
        # 归一化观测值
        # ...
        # 归一化动作
        # ...
        return new_transition

```

完整实现请参见 `src/lerobot/processor/normalize_processor.py`。

**关键原则：**

- **始终使用 `transition.copy()`** 以避免副作用
- **一致地处理观测值和动作**
- **将配置与状态分离**：`get_config()` 返回 JSON 可序列化的参数，`state_dict()` 返回张量
- **在 `__post_init__()` 中将统计信息转换为张量**，以提高计算效率

### 配置与状态管理

处理器通过三个方法支持序列化，将配置与张量状态分离。`NormalizerProcessorStep` 完美地展示了这一点——它在状态中携带数据集统计信息（张量），在配置中携带超参数：

```python
# 继续 NormalizerProcessorStep 示例...

def get_config(self) -> dict[str, Any]:
    """JSON 可序列化配置（不包含张量）。"""
    return {
        "eps": self.eps,
        "features": {k: {"type": v.type.value, "shape": v.shape} for k, v in self.features.items()},
        "norm_map": {ft.value: nm.value for ft, nm in self.norm_map.items()},
        # ...
    }

def state_dict(self) -> dict[str, torch.Tensor]:
    """仅包含张量状态（例如，数据集统计信息）。"""
    flat: dict[str, torch.Tensor] = {}
    for key, sub in self._tensor_stats.items():
        for stat_name, tensor in sub.items():
            flat[f"{key}.{stat_name}"] = tensor.cpu()  # 始终保存到 CPU
    return flat

def load_state_dict(self, state: dict[str, torch.Tensor]) -> None:
    """在运行时恢复张量状态。"""
    self._tensor_stats.clear()
    for flat_key, tensor in state.items():
        key, stat_name = flat_key.rsplit(".", 1)
        # 加载到处理器配置的设备上
        self._tensor_stats.setdefault(key, {})[stat_name] = tensor.to(
            dtype=torch.float32, device=self.device
        )
        # ...
```

**用法：**

```python
# 保存（例如，在策略内部）
config = normalizer.get_config()
tensors = normalizer.state_dict()

# 恢复（例如，加载预训练策略）
new_normalizer = NormalizerProcessorStep(**config)
new_normalizer.load_state_dict(tensors)
# 现在 new_normalizer 拥有相同的统计信息和配置
```

### 转换特征（Transform Features）

`transform_features` 方法定义了你的处理器如何转换特征名称和形状。这对于策略配置和调试至关重要。

对于 `NormalizerProcessorStep`，由于归一化不会改变键名或形状，特征通常保持不变：

```python
def transform_features(self, features: dict[PipelineFeatureType, dict[str, PolicyFeature]]) -> dict[PipelineFeatureType, dict[str, PolicyFeature]]:
    """归一化保留所有特征定义。"""
    return features  # 特征结构无变化
    # ...
```

当你的处理器重命名或重塑数据时，应实现此方法以反映对下游组件的映射关系。例如，一个简单的重命名处理器：

```python
def transform_features(self, features: dict[str, PolicyFeature]) -> dict[str, PolicyFeature]:
    # 简单重命名
    if "pixels" in features:
        features["observation.image"] = features.pop("pixels")

    # 基于模式的重命名
    for key in list(features.keys()):
        if key.startswith("env_state."):
            suffix = key[len("env_state."):]
            features[f"observation.{suffix}"] = features.pop(key)
            # ...

    return features
```

**关键原则：**

- 使用 `features.pop(old_key)` 来移除并获取旧特征
- 使用 `features[new_key] = old_feature` 来添加重命名后的特征
- 始终返回修改后的特征字典
- 在文档字符串中清晰地记录转换内容

### 使用覆盖（Overrides）

你可以在加载时使用 `overrides` 覆盖步骤参数。这对于非可序列化对象或特定于站点的设置非常方便。它既适用于策略工厂，也适用于 `DataProcessorPipeline.from_pretrained(...)`。

**基础模型适配**：这在使用基础预训练策略时特别有用，因为通常无法访问原始训练统计信息。你可以注入自己的数据集统计信息，使归一化器适应你的特定机器人或环境数据。

示例：在机器人上评估策略时，覆盖设备和重命名映射。
用于在仅 CPU 的机器人上运行在 CUDA 上训练的策略，或者当机器人使用与数据集不同的摄像头键名时进行键映射。

直接与 `from_pretrained` 一起使用：

```python
from lerobot.processor import RobotProcessorPipeline

# 加载在多种机器人数据上训练的基础策略
# 但使归一化适应你的特定机器人/环境
new_stats = LeRobotDataset(repo_id="username/my-dataset").meta.stats
processor = RobotProcessorPipeline.from_pretrained(
    "huggingface/foundational-robot-policy",  # 预训练基础模型
    overrides={
        "normalizer_processor": {"stats": new_stats},     # 注入你的机器人统计信息
        "device_processor": {"device": "cuda:0"},         # 已注册步骤的注册名称
        "rename_processor": {"rename_map": robot_key_map}, # 映射你的机器人观测键名
        # ...
    },
)
```

## 最佳实践

基于对 LeRobot 所有处理器实现的分析，以下是关键模式和实践：

### 1. **安全数据处理**

始终创建输入数据的副本以避免意外的副作用。使用 `transition.copy()` 和 `observation.copy()`，而不是就地修改数据。这可以防止你的处理器意外影响管道中的其他组件。

在处理之前检查所需数据，并优雅地处理缺失数据。如果你的处理器期望某些键名（如图像处理的 `"pixels"`），请首先验证它们的存在。对于可选数据，使用安全访问模式如 `transition.get()`，并妥善处理 `None` 值。

当数据验证失败时，提供清晰、可操作的错误消息，帮助用户理解问题所在以及如何修复。

### 2. **选择合适的基类**

LeRobot 提供了专门的基类，可以减少样板代码并确保一致性。当你只需要修改观测值时使用 `ObservationProcessorStep`，仅处理动作时使用 `ActionProcessorStep`，专门处理基于字典的机器人动作时使用 `RobotActionProcessorStep`。

只有当你需要对整个转换具有完全控制权，或同时处理多个转换组件时，才直接从 `ProcessorStep` 继承。专门的基类会为你处理转换管理并提供类型安全。

### 3. **注册与命名**

使用 `@ProcessorStepRegistry.register()` 注册带有描述性、命名空间化名称的处理器。使用组织前缀如 `"robotics_lab/safety_clipper"` 或 `"acme_corp/vision_enhancer"` 以避免命名冲突。避免使用如 `"processor"` 或 `"step"` 等通用名称，以免与其他实现冲突。

良好的注册使你的处理器可被发现，并在保存和加载管道时实现清晰的序列化/反序列化。

### 4. **状态管理模式**

区分配置参数（JSON 可序列化值）和内部状态（张量、缓冲区）。使用带有 `init=False, repr=False` 的数据类字段来表示不应出现在构造函数或字符串表示中的内部状态。

实现 `reset()` 方法以在片段之间清除内部状态。这对于随时间累积数据的有状态处理器（如移动平均或时间滤波器）至关重要。

请记住，`get_config()` 应仅返回 JSON 可序列化的配置，而 `state_dict()` 则单独处理张量状态。

### 5. **输入验证与错误处理**

在处理之前验证输入类型和形状。检查张量属性如 `dtype` 和维度，以确保与你的算法兼容。对于机器人动作，验证所需的姿态组件或关节值是否存在且在预期范围内。

对于无需处理的情况，使用提前返回来避免不必要的操作。提供清晰、描述性的错误消息，包括预期与实际的数据类型或形状。这可以大大简化用户的调试工作。

### 6. **设备与数据类型感知**

将处理器设计为自动适应输入张量的设备和数据类型。内部张量（如归一化统计信息）应与输入张量的设备和数据类型匹配，以确保与多 GPU 训练、混合精度和分布式设置兼容。

实现 `to()` 方法，将处理器的内部状态移动到指定设备。在运行时检查设备/数据类型兼容性，并在需要时自动迁移内部状态。这种模式能够在无需手动干预的情况下，在不同硬件配置之间实现无缝运行。

## 总结

现在你已经拥有了在 LeRobot 中实现自定义处理器的所有工具！关键步骤如下：

1. **定义处理器**：作为数据类，包含所需方法（`__call__`、`get_config`、`state_dict`、`load_state_dict`、`reset`、`transform_features`）
2. **注册它**：使用 `@ProcessorStepRegistry.register("name")` 以实现可发现性
3. **集成它**：将处理器与其他步骤一起集成到 `DataProcessorPipeline` 中
4. **使用基类**：尽可能使用 `ObservationProcessorStep` 等以减少样板代码
5. **实现设备/数据类型感知**：以支持多 GPU 和混合精度设置

处理器系统设计为模块化和可组合的，使你能够从简单、专注的组件构建复杂的数据处理管道。无论是为训练预处理传感器数据，还是为机器人执行后处理模型输出，自定义处理器都能为你提供处理机器人应用所需任何数据转换的灵活性。

健壮处理器的关键原则：

- **设备/数据类型自适应**：内部张量应与输入张量匹配
- **清晰的错误消息**：帮助用户理解问题所在
- **基类使用**：利用专门的基类减少样板代码
- **特征契约**：使用 `transform_features()` 声明数据结构变化

从简单开始，充分测试，并确保你的处理器在不同硬件配置之间无缝运行！
