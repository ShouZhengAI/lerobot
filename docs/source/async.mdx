# 异步推理（Asynchronous Inference）

通过我们的 [SmolVLA](https://huggingface.co/papers/2506.01844)，我们引入了一种在真实机器人上运行推理的新方法——**将动作预测与动作执行分离**。
在本教程中，我们将展示如何使用 SmolVLA 的微调版本以及 LeRobot 支持的所有策略进行异步推理（async inference）。
**尝试与 LeRobot 支持的所有策略**一起使用异步推理！

**你将学到：**

1. 为什么异步推理很重要，以及它与更传统的顺序推理相比有何优势。
2. 如何在同一台机器上启动 `PolicyServer` 并连接 `RobotClient`，甚至通过网络连接。
3. 如何根据你的机器人和策略调整关键参数（`actions_per_chunk`、`chunk_size_threshold`）。

如果遇到问题，欢迎加入我们的 [Discord 社区](https://discord.gg/s3KuuzsPFb)！

简而言之：使用_异步推理_，当策略服务器已经忙于计算下一个动作块时，你的机器人继续执行动作——消除了"等待推理"的延迟，实现了更流畅、更灵敏的行为。
这与同步推理（sync）有本质区别——同步推理中，机器人在策略计算下一个动作块期间处于空闲状态。

## 异步推理入门

你可以在我们的[博文](https://huggingface.co/blog/async-robot-inference)中阅读更多关于异步推理的信息。本指南旨在帮助你快速在环境中设置和运行异步推理。

首先，安装带有 `async` 标签的 `lerobot`，以安装运行异步推理所需的额外依赖。

```shell
pip install -e ".[async]"
```

然后，启动一个策略服务器（在一个终端中，或在独立的机器上），指定客户端连接的主机地址和端口。
启动策略服务器的命令如下：

```shell
python -m lerobot.async_inference.policy_server \
     --host=127.0.0.1 \
     --port=8080
```

这将启动一个策略服务器，监听 `127.0.0.1:8080`（localhost，端口 8080）。此时策略服务器为空，所有与运行哪个策略以及使用哪些参数相关的信息都在与客户端的首次握手期间指定。启动客户端：

```shell
python -m lerobot.async_inference.robot_client \
    --server_address=127.0.0.1:8080 \ # 服务器端：策略服务器的主机地址和端口
    --robot.type=so100_follower \ # 机器人：你的机器人类型
    --robot.port=/dev/tty.usbmodem585A0076841 \ # 机器人：你的机器人端口
    --robot.id=follower_so100 \ # 机器人：你的机器人 ID，用于加载校准文件
    --robot.cameras="{ laptop: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}, phone: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}}" \ # 策略：用于获取帧的相机，键名须与策略期望的键名匹配
    --task="dummy" \ # 策略：策略要运行的任务（如"折叠我的T恤"）。并非所有策略都需要定义，如 `act`
    --policy_type=your_policy_type \ # 策略：要运行的策略类型（smolvla、act 等）
    --pretrained_name_or_path=user/model \ # 策略：服务器上要运行的检查点的模型名称/路径（例如 lerobot/smolvla_base）
    --policy_device=mps \ # 策略：在服务器上运行策略的设备（cuda、mps、xpu、cpu）
    --actions_per_chunk=50 \ # 策略：一次输出的动作数量
    --chunk_size_threshold=0.5 \ # 客户端：触发向服务器发送新观测的动作块大小阈值
    --aggregate_fn_name=weighted_average \ # 客户端：在重叠部分聚合动作的函数
    --debug_visualize_queue_size=True # 客户端：是否在运行时可视化队列大小
```

总结来说，你需要为以下方面指定参数：

- `SERVER`（服务器端）：策略服务器的主机地址和端口
- `ROBOT`（机器人）：要连接的机器人类型、连接端口和本地机器人 `id`
- `POLICY`（策略）：要运行的策略类型以及服务器上要运行的检查点的模型名称/路径。你还需要指定服务器使用的设备以及一次输出的动作数量（上限为策略的最大动作值）。
- `CLIENT`（客户端）：触发向服务器发送新观测的动作块大小阈值，以及在重叠部分聚合动作的函数。可选地，你还可以在运行时可视化队列大小，以帮助你调整 `CLIENT` 参数。

请务必注意：

- `actions_per_chunk` 和 `chunk_size_threshold` 是需要根据你的配置进行调整的关键参数。
- `aggregate_fn_name` 是在重叠部分聚合动作的函数。你可以在函数注册表中添加新函数，也可以在 `robot_client.py` 中添加自己的函数。
- `debug_visualize_queue_size` 是调整 `CLIENT` 参数的有用工具。

## 完成！现在你应该能看到机器人在运动了😉

## 异步推理 vs. 同步推理

同步推理依赖于动作块预测和动作执行的交替进行。这本质上导致了_空闲帧_——机器人在等待策略输出新的动作块时空闲等待的帧。
因此，推理过程中存在明显的实时延迟，机器人因缺少可用动作而停止运动。随着机器人模型规模的增长，这个问题只会变得更加严重。

    width="80%"
  ></img>
</p>
<p align="center">
  <i>同步推理</i>使得机器人在策略计算下一个动作块期间处于空闲状态。
</p>

为了克服这个问题，我们设计了异步推理——一种将动作规划与执行分离的范式，实现（1）更高的适应性和（2）最重要的是没有空闲帧。
关键在于，异步推理会在当前动作块_耗尽之前_计算下一个动作块，从而消除空闲。通过在重叠部分聚合不同的动作块，可以获得最新的计划和更紧凑的控制循环，从而确保更高的适应性。

<p align="center">
  <i>异步推理</i>不会产生空闲，因为下一个动作块在当前动作块耗尽之前就已经计算完成。

## 启动策略服务器

策略服务器是 `PreTrainedPolicy` 的包装器，负责将来自机器人客户端的观测与策略连接起来。
策略服务器初始化为空容器，在机器人客户端与策略服务器之间的首次握手中根据请求的策略进行填充。
因此，启动策略服务器只需指定主机地址和端口即可。如果在与机器人客户端相同的机器上运行策略服务器，可以使用 `localhost` 作为主机地址。

<hfoption id="命令行">

```bash
     --host=127.0.0.1 \
     --port=8080
```

## 启动机器人客户端

`RobotClient` 是 `Robot` 实例的包装器，它连接到（可能是远程的）`PolicyServer`。
`RobotClient` 将观测流式传输到 `PolicyServer`，并接收在服务器上运行推理（假设服务器比机器人控制器拥有更好的计算资源）得到的动作块。

<hfoptions id="start_robot_client">
<hfoption id="命令行">
```bash
python -m lerobot.async_inference.robot_client \
    --server_address=127.0.0.1:8080 \ # 服务器端：策略服务器的主机地址和端口
    --robot.type=so100_follower \ # 机器人：你的机器人类型
    --robot.port=/dev/tty.usbmodem585A0076841 \ # 机器人：你的机器人端口
    --robot.id=follower_so100 \ # 机器人：你的机器人 ID，用于加载校准文件
    --robot.cameras="{ laptop: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}, phone: {type: opencv, index_or_path: 0, width: 1920, height: 1080, fps: 30}}" \ # 策略：用于获取帧的相机，键名须与策略期望的键名匹配
    --task="dummy" \ # 策略：策略要运行的任务（如"折叠我的T恤"）。并非所有策略都需要定义，如 `act`
    --policy_type=your_policy_type \ # 策略：要运行的策略类型（smolvla、act 等）
    --pretrained_name_or_path=user/model \ # 策略：服务器上要运行的检查点的模型名称/路径（例如 lerobot/smolvla_base）
    --policy_device=mps \ # 策略：在服务器上运行策略的设备
    --actions_per_chunk=50 \ # 策略：一次输出的动作数量
    --chunk_size_threshold=0.5 \ # 客户端：触发向服务器发送新观测的动作块大小阈值
    --aggregate_fn_name=weighted_average \ # 客户端：在重叠部分聚合动作的函数
    --debug_visualize_queue_size=True # 客户端：是否在运行时可视化队列大小
```
</hfoption>
<hfoption id="API 示例">

<!-- prettier-ignore-start -->
```python
import threading
from lerobot.robots.so_follower import SO100FollowerConfig
from lerobot.cameras.opencv import OpenCVCameraConfig
from lerobot.async_inference.configs import RobotClientConfig
from lerobot.async_inference.robot_client import RobotClient
from lerobot.async_inference.helpers import visualize_action_queue_size

# 1. 创建机器人实例
"""运行 `python lerobot/find_cameras.py` 查看你的配置中可用的相机"""
# 这些相机必须与策略期望的相机匹配
# 查看你所用策略在 Hub 上的 config.json
camera_cfg = {
    "top": OpenCVCameraConfig(index_or_path=0, width=640, height=480, fps=30),
    "side": OpenCVCameraConfig(index_or_path=1, width=640, height=480, fps=30)
}

robot_cfg = SO100FollowerConfig(
  port="/dev/tty.usbmodem585A0076841",
  id="follower_so100",
  cameras=camera_cfg
)

# 3. 创建客户端配置
client_cfg = RobotClientConfig(
    robot=robot_cfg,
    server_address="localhost:8080",
    policy_device="mps",
    client_device="cpu",
    policy_type="smolvla",
    pretrained_name_or_path="<user>/smolvla_async",
    chunk_size_threshold=0.5,
    actions_per_chunk=50,  # 确保此值小于策略的最大动作数
)

# 4. 创建并启动客户端
client = RobotClient(client_cfg)

# 5. 指定任务
task = "不要做任何动作，保持静止"

if client.start():
    # 启动动作接收线程
    action_receiver_thread = threading.Thread(target=client.receive_actions, daemon=True)
    action_receiver_thread.start()

    try:
        # 运行控制循环
        client.control_loop(task)
    except KeyboardInterrupt:
        client.stop()
        action_receiver_thread.join()
        # （可选）绘制动作队列大小
        visualize_action_queue_size(client.action_queue_size)
```

<Tip>
  `actions_per_chunk` 和 `chunk_size_threshold` 的不同取值确实会导致不同的行为。
</Tip>

一方面，增大 `actions_per_chunk` 的值会降低动作消耗完毕的可能性，因为计算新动作块时将会有更多的可用动作。
然而，较大的 `actions_per_chunk` 值也可能导致动作精确度降低，因为预测跨更长时间段的动作会积累复合误差。

另一方面，增大 `chunk_size_threshold` 的值会导致更频繁地向 `PolicyServer` 发送观测进行推理，从而获得更多的更新动作块，并在显著部分上形成重叠。这带来了更高的适应性，理论上可以达到每个观测预测一个动作块，该动作块仅被部分消耗时新的一块就已经生产出来。
该选项也会给推理管道带来更大的压力，因为请求数量增加了。相反，接近 0.0 的 `chunk_size_threshold` 值会退化到同步的边缘情况——只有当前动作块耗尽时才发送新的观测。

我们发现 `actions_per_chunk` 和 `chunk_size_threshold` 的默认值在 [SmolVLA 论文](https://huggingface.co/papers/2506.01844)的实验中使用效果良好，但建议尝试不同的值以找到最适合你配置的方案。

### 为你的配置调整异步推理

1. **仔细选择你的计算资源。** [PI0](https://huggingface.co/lerobot/pi0) 推理时占用 14GB 内存，而 [SmolVLA](https://huggingface.co/lerobot/smolvla_base) 仅需约 2GB。你应该根据使用场景确定最佳的计算资源，记住较小的策略需要较少的计算资源。策略和使用设备（CPU密集型、使用 MPS 或特定 NVIDIA GPU 上的 CUDA 核心数）的组合直接影响你应预期的平均推理延迟。
2. **根据推理延迟调整 `fps`。** 服务器生成新的动作块时，客户端并非空闲，而是在执行当前的动作队列。如果两个进程的速度存在根本性差异，客户端可能会耗尽队列中的动作。因此，如果队列中的动作持续运行完毕，应降低 fps。
3. **调整 `chunk_size_threshold`。**
   - 接近 `0.0` 的值导致几乎顺序执行的行为。接近 `1.0` 的值则每一步都发送观测（带宽更高，依赖于良好的世界模型）。
   - 我们发现 0.5-0.6 左右的值效果很好。如果你想调整此参数，可以启动一个 `RobotClient` 并将 `--debug_visualize_queue_size` 设置为 `True`。这将在运行时绘制动作队列大小的变化，你可以利用它找到最适合你的配置的 `chunk_size_threshold` 值。

## 结论

异步推理代表了实时机器人控制领域的重大进步，解决了长期困扰机器人应用的推理延迟这一根本挑战。通过本教程，你已学会如何实现一个完整的异步推理管道，消除空闲帧，实现更流畅、更灵敏的机器人行为。

**关键要点：**

- **范式转变**：异步推理将动作预测与执行分离，使机器人在并行计算新动作块的同时继续执行动作
- **性能优势**：消除了同步方法中固有的"等待推理"延迟，随着策略模型不断增大这一点变得愈发重要
- **灵活的架构**：服务器-客户端设计支持分布式计算，推理可以在强大的远程硬件上运行，同时保持实时机器人控制
- **可调参数**：成功取决于根据你的具体硬件、策略和任务要求正确配置 `actions_per_chunk` 和 `chunk_size_threshold`
- **通用兼容性**：适用于 LeRobot 支持的所有策略，从轻量级的 ACT 模型到 SmolVLA 等视觉-语言模型

开始使用默认参数进行实验，监控动作队列大小，并迭代优化你的配置以达到特定使用场景的最佳性能。
