# 添加策略

本指南将引导你实现一个自定义策略并使其与 LeRobot 的训练、评估和部署工具协同工作。有两种路径可供选择：

- **插件式（树外）** —— 将你的策略作为一个独立的 `lerobot_policy_*` 包发布。速度更快，无需 PR，便于迭代。适合实验研究、内部使用，或者希望独立发布时使用。
- **树内式（贡献到 LeRobot 中）** —— 将你的策略直接放入 `src/lerobot/policies/`。需要提交 PR，但能让你的策略成为库的一等公民。

插件路径通常是合适的起点 —— 待策略稳定且确实有价值与库一起发布时，再升级为树内式。

无论哪种方式，构建模块都是一样的：一个配置类、一个策略类和一个处理器工厂。本指南的前半部分涵盖这些共享组件；后半部分介绍路径特定的脚手架（[路径 A：树外插件](#路径-a树外插件)、[路径 B：贡献到树内](#路径-b贡献到树内)）。

关于语气的一点说明：机器人学习是一个活跃演进的领域，"策略应该是什么样"会随着每个新架构而变化。这里描述的约定之所以存在，是因为它们让 `lerobot-train` 和 `lerobot-eval` 能够在非常不同的模型上统一工作。当一个新的策略确实无法适应这些约定时，请提出来（在你的 PR 或 issue 中）—— 这些约定并非神圣不可侵犯。

---

## 策略的组成部分

每个策略由三个构建模块组成。下面的名称使用 `my_policy` 作为占位符 —— 请替换为你的策略名称。这个名称是关键的：它必须与你传递给 `@PreTrainedConfig.register_subclass` 的字符串、`MyPolicy.name` 类属性以及 `make_<name>_pre_post_processors` 工厂函数名一致（下文将详细介绍每一项）。

### 配置类

继承 [`PreTrainedConfig`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/configs/policies.py) 并注册你的策略类型。以下是一个模板 —— 根据你的策略架构和训练需求自定义参数和方法。

```python
# configuration_my_policy.py
from dataclasses import dataclass, field
from lerobot.configs import PreTrainedConfig
from lerobot.optim import AdamWConfig
from lerobot.optim import CosineDecayWithWarmupSchedulerConfig

@PreTrainedConfig.register_subclass("my_policy")
@dataclass
class MyPolicyConfig(PreTrainedConfig):
    """MyPolicy 的配置类。

    Args:
        n_obs_steps: 用作输入的观测步数
        horizon: 动作预测范围
        n_action_steps: 执行的动作步数
        hidden_dim: 策略网络的隐藏层维度
        # 在此处添加你的策略专属参数
    """

    horizon: int = 50
    n_action_steps: int = 50
    hidden_dim: int = 256

    optimizer_lr: float = 1e-4
    optimizer_weight_decay: float = 1e-4

    def __post_init__(self):
        super().__post_init__()
        if self.n_action_steps > self.horizon:
            raise ValueError("n_action_steps 不能超过 horizon")

    def validate_features(self) -> None:
        """验证输入/输出特征的兼容性。

        在你的策略的 __init__ 中显式调用此方法 —— 基类不会自动调用。
        """
        if not self.image_features:
            raise ValueError("MyPolicy 至少需要一个图像特征。")
        if self.action_feature is None:
            raise ValueError("MyPolicy 需要在 output_features 中包含 'action'。")

    def get_optimizer_preset(self) -> AdamWConfig:
        return AdamWConfig(lr=self.optimizer_lr, weight_decay=self.optimizer_weight_decay)

    def get_scheduler_preset(self):
        """从 lerobot.optim 返回一个 LRSchedulerConfig，或返回 None。"""
        return None

    @property
    def observation_delta_indices(self) -> list[int] | None:
        """数据集加载器为每个观测提供的相对时间步偏移。

        对于单帧策略返回 `None`。对于消费多个过去或未来帧的时序策略，
        返回偏移列表，例如 `[-20, -10, 0, 10]` 表示步长为 10 的 3 个过去帧和
        1 个未来帧。
        """
        return None

    @property
    def action_delta_indices(self) -> list[int]:
        """数据集加载器返回的动作块（action chunk）的相对时间步偏移。"""
        return list(range(self.horizon))

    @property
    def reward_delta_indices(self) -> None:
        return None
```

传递给 `@register_subclass` 的字符串必须与 `MyPolicy.name`（下一节）一致，并且是用户在 CLI 上通过 `--policy.type` 提供的值。除非确实需要其他优化器，否则 `get_optimizer_preset` 默认使用 `lerobot.optim` 中的 `AdamW`。

### 策略类

继承 [`PreTrainedPolicy`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/pretrained.py) 并设置两个类属性 —— `__init_subclass__` 会检查这两项：

```python
# modeling_my_policy.py
import torch
import torch.nn as nn
from typing import Any

from lerobot.policies import PreTrainedPolicy
from lerobot.utils.constants import ACTION
from .configuration_my_policy import MyPolicyConfig

class MyPolicy(PreTrainedPolicy):
    config_class = MyPolicyConfig  # 必须与 @register_subclass 中的字符串一致
    name = "my_policy"

    def __init__(self, config: MyPolicyConfig, dataset_stats: dict[str, Any] = None):
        super().__init__(config, dataset_stats)
        config.validate_features()  # 基类不会自动调用
        self.config = config
        self.model = ...  # 在此处定义你的 nn.Module

    def reset(self):
        """重置每 episode 的状态。lerobot-eval 在每个 episode 开始时调用此方法。"""
        ...

    def get_optim_params(self) -> dict:
        """返回传递给优化器的参数（例如带分组学习率/权重衰减）。"""
        return {"params": self.parameters()}

    def predict_action_chunk(self, batch: dict[str, torch.Tensor], **kwargs) -> torch.Tensor:
        """返回当前观测的完整动作块 (B, chunk_size, action_dim)。"""
        ...

    def select_action(self, batch: dict[str, torch.Tensor], **kwargs) -> torch.Tensor:
        """返回当前时间步的单个动作（推理时每一步调用）。"""
        ...

    def forward(self, batch: dict[str, torch.Tensor]) -> tuple[torch.Tensor, dict | None]:
        """计算训练损失。

        返回 `(loss, output_dict)`。`output_dict` 可以为 `None`；
        其中的所有内容必须是日志友好的 Python 原生类型（不带梯度的张量）。

        `batch["action_is_pad"]` 是一个形状为 (B, horizon) 的布尔掩码，
        标记因 episode 在 `horizon` 步之前结束而被填充的时间步；
        你可以将这些时间步排除在损失计算之外。
        """
        actions = batch[ACTION]
        action_is_pad = batch.get("action_is_pad")
        ...
        return loss, {"some_loss_component": some_loss_component.item()}
```

训练/评估循环调用的方法：

| 方法                                                               | 使用方           | 功能                                                                                                                                                                                                                                                                                          |
| ------------------------------------------------------------------ | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `reset() -> None`                                                  | `lerobot-eval`   | 在每个 episode 开始时清除每 episode 状态。                                                                                                                                                                                                                                                     |
| `select_action(batch, **kwargs) -> Tensor`                         | `lerobot-eval`   | 返回下一个动作 `(B, action_dim)`。每一步调用。                                                                                                                                                                                                                                                |
| `predict_action_chunk(batch, **kwargs) -> Tensor`                  | 策略自身         | 返回动作块 `(B, chunk_size, action_dim)`。当前在基类上是抽象的 —— 如果你的策略不使用分块，请抛出 `NotImplementedError`。                                                                                                                                                                         |
| `forward(batch, reduction="mean") -> tuple[Tensor, dict \| None]`  | `lerobot-train`  | 返回 `(loss, output_dict)`。如果要支持按样本加权，接受 `reduction="none"`。                                                                                                                                                                                                                  |
| `get_optim_params() -> dict`                                       | 优化器           | 简单策略返回 `self.parameters()`；多优化器策略返回命名参数字典（参见 [`modeling_act.py`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/act/modeling_act.py) 中的 `get_optim_params` 以了解按分组学习率的示例）。                                                         |
| `update() -> None` _(可选)_                                        | `lerobot-train`  | 如果定义了此方法，则在每个优化器步骤后调用。用于 EMA、目标网络、回放缓冲区（TDMPC 使用此方法）。                                                                                                                                                                                              |

批次（batch）是以 [`lerobot.utils.constants`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/utils/constants.py) 中的常量为键的扁平字典：`OBS_STATE`（`observation.state.<motor>`）、`OBS_IMAGES`（`observation.images.<camera>`）、`OBS_LANGUAGE`、`ACTION` 等。请复用这些常量 —— 不要发明新的前缀。

### 处理器函数

LeRobot 使用 `PolicyProcessorPipeline` 对策略的输入进行归一化，对输出进行反归一化。具体参考请参见 [`processor_act.py`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/act/processor_act.py) 或 [`processor_diffusion.py`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/diffusion/processor_diffusion.py)。

```python
# processor_my_policy.py
from typing import Any
import torch

from lerobot.processor import PolicyAction, PolicyProcessorPipeline


def make_my_policy_pre_post_processors(
    config,
    dataset_stats: dict[str, dict[str, torch.Tensor]] | None = None,
) -> tuple[
    PolicyProcessorPipeline[dict[str, Any], dict[str, Any]],
    PolicyProcessorPipeline[PolicyAction, PolicyAction],
]:
    preprocessor = ...   # 为输入构建你的 PolicyProcessorPipeline
    postprocessor = ...  # 为输出构建你的 PolicyProcessorPipeline
    return preprocessor, postprocessor
```

**重要 —— 函数命名：** LeRobot 通过名称发现处理器。函数**必须**命名为 `make_{policy_name}_pre_post_processors`（与你传递给 `@PreTrainedConfig.register_subclass` 的字符串一致）。

## 路径 A：树外插件

发布策略最快的方式：将其打包为独立的 Python 发行版并安装到 LeRobot 旁边。无需 PR，你可以自行掌控发布周期，并且可以在自己的命名空间下发布到 PyPI。

### 包结构

创建一个以 `lerobot_policy_` 为前缀（重要！）后接策略名称的包：

```bash
lerobot_policy_my_policy/
├── pyproject.toml
└── src/
    └── lerobot_policy_my_policy/
        ├── __init__.py
        ├── configuration_my_policy.py
        ├── modeling_my_policy.py
        └── processor_my_policy.py
```

### `pyproject.toml`

```toml
[project]
name = "lerobot_policy_my_policy"
version = "0.1.0"
dependencies = [
    # 你的策略专属依赖
]
requires-python = ">= 3.12"

[build-system]
build-backend = # 你的构建后端
requires = # 你的构建系统
```

### 包的 `__init__.py`

在包的 `__init__.py` 中暴露你的类，并对缺失 `lerobot` 的情况进行检查：

```python
# __init__.py
"""用于 LeRobot 的自定义策略包。"""

try:
    import lerobot  # noqa: F401
except ImportError:
    raise ImportError(
        "未安装 lerobot。请先安装 lerobot 再使用此策略包。"
    )

from .configuration_my_policy import MyPolicyConfig
from .modeling_my_policy import MyPolicy
from .processor_my_policy import make_my_policy_pre_post_processors

__all__ = [
    "MyPolicyConfig",
    "MyPolicy",
    "make_my_policy_pre_post_processors",
]
```

### 安装和使用

```bash
cd lerobot_policy_my_policy
pip install -e .

# 或者如果已发布到 PyPI 则从 PyPI 安装
pip install lerobot_policy_my_policy
```

安装后，你的策略会自动集成到 LeRobot 的训练和评估工具中：

```bash
lerobot-train \
    --policy.type my_policy \
    --env.type pusht \
    --steps 200000
```

## 路径 B：贡献到树内

当你的策略已经稳定且确实有价值与库一起发布时，你可以将其直接放入 LeRobot。请先阅读通用的[贡献指南](./contributing)和 [PR 模板](https://github.com/huggingface/lerobot/blob/main/.github/PULL_REQUEST_TEMPLATE.md) —— 那里会找到每个 PR 都需要满足的测试/质量期望（`pre-commit run -a`、`pytest`、社区评审规则等）。下面是在此基础上的策略特定层。

### 树内布局

```
src/lerobot/policies/my_policy/
├── __init__.py                    # 重新导出配置 + 建模 + 处理器工厂
├── configuration_my_policy.py     # MyPolicyConfig + @register_subclass
├── modeling_my_policy.py          # MyPolicy(PreTrainedPolicy)
├── processor_my_policy.py         # make_my_policy_pre_post_processors
└── README.md                      # 符号链接 → ../../../../docs/source/policy_my_policy_README.md
```

两点说明：

- 源码旁边的 `README.md` 是指向 `docs/source/policy_<name>_README.md` 的**符号链接** —— 实际文件位于 `docs/` 下。现有策略（act、smolvla、diffusion 等）都采用这种方式；复制其中一个符号链接即可。策略 README 通常保持极简：论文链接 + BibTeX 引用。
- 面向用户的教程 —— 安装什么、如何训练、超参数、基准测试数据 —— 单独位于 `docs/source/<my_policy>.mdx` 中，并在 `_toctree.yml` 的"Policies"下注册。

文件名是关键的：工厂通过名称进行惰性导入，处理器通过 `make_<policy_name>_pre_post_processors` 约定被发现。

### 接入

有两个地方需要知道你的策略。全部通过名称引用。

1. **`policies/__init__.py`** —— 重新导出 `MyPolicyConfig` 并将其添加到 `__all__` 中。这个导入会注册你的策略：`@PreTrainedConfig.register_subclass("my_policy")` 执行，之后工厂通过约定解析所有内容。**不要**重新导出建模类；它会通过工厂惰性加载（这样 `import lerobot` 就能保持快速）。
2. **`templates/lerobot_modelcard_template.md` 和根目录 `README.md`** —— 该模板是 `push_model_to_hub` 用来渲染用你的策略训练的每个检查点的模型卡片的内容：在 `model_name` 分支中添加一行你的策略描述，在 `policy_docs` 中映射以便卡片链接到你的 MDX 指南，并可选地在 `diagrams` 中添加架构图片。然后在根目录 `README.md` 的模型表中将你的策略添加到相应类别下，链接到你的文档页面。

参考一个与你结构相似的现有策略；差异很小。

### 重量级 / 可选依赖

大多数策略需要重量级骨干网络（transformers、diffusers、特定的 VLM SDK）。约定是**两步守卫**：在模块顶部使用 `TYPE_CHECKING` 保护的导入，在构造函数中使用 `require_package` 运行时检查。[`modeling_diffusion.py`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/diffusion/modeling_diffusion.py) 是规范的参考：

```python
from typing import TYPE_CHECKING
from lerobot.utils.import_utils import _diffusers_available, require_package

if TYPE_CHECKING or _diffusers_available:
    from diffusers.schedulers.scheduling_ddim import DDIMScheduler
else:
    DDIMScheduler = None  # 在导入时保持符号可绑定

class DiffusionPolicy(PreTrainedPolicy):
    def __init__(self, config):
        require_package("diffusers", extra="diffusion")
        super().__init__(config)
        ...
```

这样做的好处是：

- `import lerobot.policies` 在未安装额外依赖的情况下仍能正常工作（符号仅绑定为 `None`）。
- 类型检查器能看到真实的符号。
- 在未安装额外依赖的情况下实例化策略会抛出一个清晰的 `ImportError`，提示 `pip install 'lerobot[diffusion]'`。

在 [`pyproject.toml`](https://github.com/huggingface/lerobot/blob/main/pyproject.toml) 的 `[project.optional-dependencies]` 中添加匹配的额外选项，并将其包含在 `all` 额外选项中，这样 `pip install 'lerobot[all]'` 就能继续安装所有内容。

### 不要复制建模文件 —— 应继承它

如果你的策略需要修改 `transformers` 中已有的骨干网络（自定义条件化、额外输入、替换的子模块），**请不要复制一份它的 `modeling_*.py`**。相反，继承最小的上游单元，只覆盖需要更改的部分。[`pi_gemma.py`](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/pi_gemma.py) 是规范的参考：它通过继承 `GemmaModel`/`PaliGemmaModel` 并重写解码器层的前向传播，用约 370 行代码向 PaliGemma/Gemma 注入了 AdaRMS 条件化，而不是复制约 2000 行的建模文件。在**已加载**的原生模型上进行模型手术也是可以的（层截断、分词器扩展、隐藏状态捕获 —— 参见 `evo1/internvl3_embedder.py`、`eo1/modeling_eo1.py`、`groot/groot_n1_7.py` 中的工作示例）。当 PR 提交了一个复制的建模文件时，审查者会要求使用这种模式；唯一接受的例外是 `transformers` 中完全不存在的模型。

### 基准测试和已发布的检查点

一个新的策略如果附带一个可工作的检查点和至少一个可复现的数据，会更容易审查 —— 也更有用得多。

**至少选择一个树内基准测试。** LeRobot 提供了带有每基准测试 Docker 镜像的模拟基准测试（LIBERO、LIBERO-plus、Meta-World、RoboTwin 2.0、RoboCasa365、RoboCerebra、RoboMME、VLABench 等）。选择与你的策略模态匹配的 —— VLA 通常使用 LIBERO 或 VLABench；纯图像 BC 使用 LIBERO 或 Meta-World。完整列表见文档侧边栏的 [Benchmarks](./libero)。

**将检查点和处理器**推送到 Hub 上 `lerobot/<policy>_<benchmark>` 下（如果没有写入权限，放在你的命名空间下；维护者可以镜像它）。使用 `PreTrainedPolicy.push_model_to_hub`，这样仓库会获得 `config.json`、`model.safetensors` 和一个模型卡片。

**在你的策略的 MDX 中报告结果**，附上精确的 `lerobot-eval` 命令和硬件，以便任何人都能重新运行：

```markdown
## 结果

使用 `lerobot/<policy>_libero` 在 LIBERO 上评估：

| 套件（Suite）   | 成功率   | 片段数 |
| --------------- | -------: | -----: |
| libero_spatial  |    87.5% |     50 |
| libero_object   |    93.0% |     50 |
| libero_goal     |    81.5% |     50 |
| libero_10       |    62.0% |     50 |
| **平均**        | **81.0%**|    200 |

复现：`lerobot-eval --policy.path=lerobot/<policy>_libero --env.type=libero --env.task=libero_spatial --eval.n_episodes=50`（1× A100 40 GB）。
```

每个套件使用 `n_episodes ≥ 50` 以获得稳定的成功率估计。

如果你的策略是纯真机机器人且没有适用的模拟基准测试，则将模拟评估替换为：Hub 上的公开训练数据集、`lerobot-train` 命令、检查点，以及通过 `lerobot-rollout --policy.path=...` 在 ≥10 个片段上获得的真机成功率。

### PR 检查清单

通用要求在 [`CONTRIBUTING.md`](https://github.com/huggingface/lerobot/blob/main/CONTRIBUTING.md) 和 [PR 模板](https://github.com/huggingface/lerobot/blob/main/.github/PULL_REQUEST_TEMPLATE.md) 中。在此之上，审查者还会检查：

- [ ] `MyPolicy` 和 `MyPolicyConfig` 覆盖了上述接口；`__init_subclass__` 接受该类。
- [ ] `policies/__init__.py` 重新导出配置（这将注册策略；工厂通过命名约定解析建模/处理器）。
- [ ] `make_my_policy_pre_post_processors` 遵循命名约定。
- [ ] 可选依赖通过 `[project.optional-dependencies]` 额外选项和 `TYPE_CHECKING + require_package` 守卫保护。
- [ ] `tests/policies/` 已更新；已提交向后兼容性工件和策略特定测试。
- [ ] `src/lerobot/policies/<name>/README.md` 符号链接到 `docs/source/policy_<name>_README.md`；已编写面向用户的 `docs/source/<name>.mdx` 并添加到 `_toctree.yml`。
- [ ] `templates/lerobot_modelcard_template.md` 为你的策略添加了描述条目和 `policy_docs` 链接。
- [ ] 根目录 `README.md` 中的模型表将你的策略列在正确的类别下，链接到你的文档页面。
- [ ] 策略 MDX 中至少有一项可复现的基准测试评估，附带已发布的检查点（模拟基准测试，或真机数据集 + 检查点）。

获得干净 PR 的最快方法是复制与你最相似的现有策略的目录，重命名，然后逐个方法替换内容。不要等到所有内容都打磨好了再提交 —— 尽早开启草稿 PR 并与我们一起迭代；审查者更愿意对半成品分支提供反馈，而不是对完全合并好的分支。
