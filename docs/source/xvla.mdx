# X-VLA：首个软提示机器人基础模型，适配任意机器人与任意任务

## 概述

多年来，机器人领域一直渴望构建能够理解人类自然语言指令、并在多种环境和机器人本体上灵活操作的智能体。大语言模型（LLM）和视觉语言模型（VLM）的最新突破指明了一条前进之路：通过将动作作为基础来扩展这些基础模型架构以实现具身控制。这催生了视觉-语言-动作（VLA，Vision-Language-Action）模型的兴起，人们期望一个通用模型能够同时具备广泛的语义理解能力和稳健的操作技能。

然而，训练这样的模型困难重重。机器人数据分散在不同的平台、传感器、本体和采集协议上。异构性无处不在：不同的机械臂配置、不同的动作空间、不同的相机设置、不同的视觉域以及不同的任务分布。这些不一致性造成了严重的分布偏移，使得预训练不稳定，迁移适应不可靠。

受元学习和提示学习（prompt learning）的启发，我们提出一个问题：**"如果 VLA 模型也能像大语言模型通过提示学习任务一样，通过提示来学习每个机器人和数据集的结构特征呢？"**

**X-VLA** 是一个基于软提示（soft-prompted）的流匹配（flow-matching）VLA 框架，它将每个硬件配置视为一个"任务"，并使用少量可学习的嵌入向量进行编码。这些**软提示（Soft Prompts）**捕获了本体和领域特定的变化，在从多模态融合的最初阶段就引导着 Transformer。借助这一机制，X-VLA 能够在单一统一架构中协调不同的机器人形态、数据类型和传感器配置。

## 训练 X-VLA

### 推荐训练配置

当针对新的本体或任务微调 X-VLA 时，我们建议不要冻结 VLM，并将 `policy.dtype=bfloat16` 以避免显存不足（OOM）错误。

```bash
lerobot-train \
  --dataset.repo_id=你的数据集 \
  --output_dir=./outputs/xvla_training \
  --job_name=xvla_training \
  --policy.path="lerobot/xvla-base" \
  --policy.repo_id="HF_USER/xvla-your-robot" \
  --policy.dtype=bfloat16 \
  --policy.action_mode=auto \
  --steps=20000 \
  --policy.device=cuda \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true \
```

### 训练参数说明

| 参数                     | 默认值    | 描述                               |
| ------------------------ | --------- | ---------------------------------- |
| `freeze_vision_encoder`  | `false`   | 不冻结 VLM 视觉编码器权重          |
| `freeze_language_encoder`| `false`   | 不冻结 VLM 语言编码器权重          |
| `train_policy_transformer`| `true`    | 允许训练策略 Transformer 层        |
| `train_soft_prompts`     | `true`    | 允许训练软提示                     |

**💡 最佳实践**：在第二阶段适应新本体时，不要冻结 VLM 编码器，同时训练策略 Transformer 和软提示。

### 示例：在双臂机器人上训练

```bash
lerobot-train \
  --dataset.repo_id=<用户名>/bimanual-so100-handover-cube \
  --output_dir=./outputs/xvla_bimanual \
  --job_name=xvla_so101_training \
  --policy.path="lerobot/xvla-base" \
  --policy.dtype=bfloat16 \
  --policy.repo_id="你的用户名/xvla-biso101" \
  --steps=3000 \
  --policy.device=cuda \
  --policy.action_mode=so101_bimanual \
  --policy.freeze_vision_encoder=false \
  --policy.freeze_language_encoder=false \
  --policy.train_policy_transformer=true \
  --policy.train_soft_prompts=true
```

💡 **最佳性能：** 如果您有充足的计算资源并希望获得最佳的 X-VLA 微调性能，应遵循官方微调策略：

**🔥 以自定义学习率方案对所有组件进行全参数微调**

为确保优化稳定，视觉语言模型必须使用基础学习率的 1/10 进行训练，而所有其他组件使用完整学习率。这个学习率比例对于实现稳健的微调性能至关重要。该配置已在默认设置中为您完成。
❕注意

完全复现官方报告的性能可能需要为软提示额外增加预热学习率调度，这可以带来小幅提升。我们建议您在自定义训练流程中实现这一点以获得最佳效果。

## 核心概念

### 1. 动作模式（Action Modes）

X-VLA 使用**动作注册表（Action Registry）**系统来处理不同的动作空间和机器人本体。`action_mode` 参数定义了动作的处理方式、使用的损失函数以及预测的后处理方式。

#### 可用的动作模式

| 动作模式          | 动作维度              | 描述                                    | 使用场景                        |
| ----------------- | --------------------- | --------------------------------------- | ------------------------------- |
| `ee6d`            | 20                    | 末端执行器，包含 xyz、6D 旋转、夹爪     | 需要空间控制的双臂设置          |
| `joint`           | 14                    | 含夹爪的关节空间                        | 直接关节控制的机器人            |
| `agibot_ee6d`     | 20                    | AGI-bot 变体，使用 MSE 损失             | AGI-bot 平台                    |
| `so101_bimanual`  | 20（模型）、12（真实）| SO101 双臂机器人                        | 双臂操作任务                    |
| `auto`            | 20（模型）、自动（真实）| 从数据集自动检测动作维度              | **推荐**用于新型机器人          |

#### 为什么动作模式很重要

当您有一个像 `lerobot/xvla-base` 这样以 `action_dim=20` 预训练的检查点，而您要在不同动作维度的数据集上训练时（例如双臂机械臂的 14 维），不能简单地裁剪动作维度。动作模式负责协调：

1. **损失计算（Loss Computation）**：不同动作组件使用不同的损失函数（关节使用 MSE，夹爪使用 BCE 等）
2. **预处理（Preprocessing）**：将夹爪通道归零、填充维度
3. **后处理（Postprocessing）**：对夹爪 logits 应用 sigmoid 函数、裁剪填充维度

#### 示例：BimanualSO101 动作空间

`so101_bimanual` 动作模式处理模型输出（20D）与真实机器人控制（12D）之间的不匹配：

```python
# 模型输出 20 维度以保持兼容性
dim_action = 20

# 真实机器人只需要 12 个维度
# [左臂 (6), 右臂 (6)] = [关节 (5) + 夹爪 (1)] × 2
REAL_DIM = 12

# 预处理：将 12D 动作填充到 20D 用于训练
# 后处理：将 20D 预测裁剪到 12D 用于部署
```

详见 [action_hub.py](https://github.com/huggingface/lerobot/blob/main/src/lerobot/policies/xvla/action_hub.py) 的实现。

#### 自动动作模式（推荐）

`auto` 动作模式是使用 X-VLA 适配任何机器人的最简单方式。它会自动检测您数据集的动作维度并处理填充/裁剪：

```bash
lerobot-train \
  --policy.path="lerobot/xvla-base" \
  --policy.action_mode=auto \
  --policy.max_action_dim=20 \
  ...
```

**工作原理：**

- 从您的数据集中读取 `action_feature.shape[-1]`（例如 Franka 为 7）
- 模型输出 `max_action_dim`（默认 20）以兼容预训练
- 损失**仅在真实维度上**计算：`MSE(pred[:,:,:real_dim], target[:,:,:real_dim])`
- 后处理将输出裁剪回 `real_dim` 用于机器人控制

这消除了为大多数机器人创建自定义动作模式的需求。

### 2. 领域 ID（Domain IDs）

领域 ID 是用于不同机器人配置和相机设置的可学习标识符。它们使 X-VLA 能够区分：

- 不同的机器人（机器人 1 与机器人 2）
- 不同的相机配置（cam1 与 cam2）
- 不同的组合（Robot1-cam1-cam2 与 Robot1-cam1 与 Robot2-cam1）

#### 设置领域 ID

**训练时**：默认情况下，通用训练的领域 ID 设置为 0。

**评估时**：指定与检查点训练配置匹配的领域 ID。

```python
# 示例：LIBERO 检查点使用 domain_id=3
domain_id = 3
```

领域 ID 由预处理流程中的 `XVLAAddDomainIdProcessorStep` 自动添加到观测数据中。

`lerobot/xvla-base` 模型在以下领域 ID 上训练过。建议选择一个最接近您机器人/配置的 ID：

### 3. 处理器步骤（Processor Steps）

X-VLA 需要特定的预处理和后处理步骤以确保正常运行。

#### 必需的预处理步骤

1. **XVLAImageToFloatProcessorStep**：将图像从 [0, 255] 范围转换到 [0, 1] 范围
2. **XVLAImageNetNormalizeProcessorStep**：应用 ImageNet 归一化（VLM 骨干网络需要）
3. **XVLAAddDomainIdProcessorStep**：将领域 ID 添加到观测数据中

#### 自定义处理器示例

对于 LIBERO 环境，一个自定义处理器处理特定的观测格式：

```python
from lerobot.policies.xvla.processor_xvla import LiberoProcessorStep

processor = LiberoProcessorStep()
# 处理 robot_state 字典，将旋转矩阵转换为 6D 表示
# 对图像应用 180 度旋转以适配相机约定
```

### 4. 配置参数

X-VLA 的关键配置参数：

```python
# 观测和动作
n_obs_steps: int = 1          # 观测时间步数
chunk_size: int = 32           # 动作序列长度
n_action_steps: int = 32       # 要执行的动作步数

# 模型架构
hidden_size: int = 1024        # Transformer 隐藏层维度
depth: int = 24                # Transformer 层数
num_heads: int = 16            # 注意力头数
num_domains: int = 30          # 最大领域 ID 数量
len_soft_prompts: int = 32     # 软提示嵌入长度

# 动作空间
action_mode: str = "ee6d"      # 动作空间类型（使用 "auto" 表示自动检测）
use_proprio: bool = True       # 是否使用本体感知状态
max_state_dim: int = 32        # 最大状态维度
max_action_dim: int = 20       # 用于填充的最大动作维度（"auto" 模式使用）

# 视觉
num_image_views: int | None    # 相机视角数量
resize_imgs_with_padding: tuple[int, int] | None  # 带填充的目标图像尺寸

# 训练
num_denoising_steps: int = 10  # 流匹配去噪步数
```

## 创建自定义动作模式

如果您的机器人具有独特的动作空间，可以创建自定义动作模式：

### 步骤 1：定义您的动作空间

```python
from lerobot.policies.xvla.action_hub import BaseActionSpace, register_action
import torch.nn as nn

@register_action("my_custom_robot")
class MyCustomActionSpace(BaseActionSpace):
    """我的机器人的自定义动作空间。"""

    dim_action = 15  # 您机器人的动作维度
    gripper_idx = (7, 14)  # 夹爪通道索引

    def __init__(self):
        super().__init__()
        self.mse = nn.MSELoss()
        self.bce = nn.BCEWithLogitsLoss()

    def compute_loss(self, pred, target):
        """定义您的损失计算。"""
        # 示例：关节使用 MSE，夹爪使用 BCE
        joints_loss = self.mse(pred[:, :, :7], target[:, :, :7])
        gripper_loss = self.bce(pred[:, :, self.gripper_idx],
                                target[:, :, self.gripper_idx])

        return {
            "joints_loss": joints_loss,
            "gripper_loss": gripper_loss,
        }

    def preprocess(self, proprio, action, mode="train"):
        """训练前对动作进行预处理。"""
        # 示例：将本体感知中的夹爪通道置零
        proprio_m = proprio.clone()
        action_m = action.clone() if action is not None else None
        proprio_m[..., self.gripper_idx] = 0.0
        if action_m is not None:
            action_m[..., self.gripper_idx] = 0.0
        return proprio_m, action_m

    def postprocess(self, action):
        """对预测结果进行后处理以用于部署。"""
        # 示例：对夹爪 logits 应用 sigmoid 函数
        action[..., self.gripper_idx] = torch.sigmoid(action[..., self.gripper_idx])
        return action
```

### 步骤 2：使用您的自定义动作模式

```bash
lerobot-train \
  --policy.action_mode=my_custom_robot \
  --dataset.repo_id=您的数据集 \
  --policy.path="lerobot/xvla-base" \
  ...
```

## 故障排除

### 常见问题

**问题**："动作维度不匹配"

- **解决方案**：检查您的 `action_mode` 是否与机器人的动作空间匹配。必要时创建自定义动作模式。

**问题**："图像值超出 [0, 1] 范围"

- **解决方案**：确保图像在归一化之前已通过 `XVLAImageToFloatProcessorStep` 进行预处理。

**问题**："未找到领域 ID"

- **解决方案**：确保预处理流程中包含 `XVLAAddDomainIdProcessorStep`，并且使用了正确的 domain_id。

**问题**："在新本体上成功率低"

- **解决方案**：
  1. 验证您的 action_mode 是否正确
  2. 检查软提示是否正在训练（`train_soft_prompts=True`）
  3. 确保预处理正确（ImageNet 归一化、领域 ID）
  4. 考虑增加训练步数

**问题**："训练时显存不足"

- **解决方案**：
  1. 减小 `chunk_size`（例如从 32 减到 16）
  2. 启用梯度检查点（gradient checkpointing）
  3. 减小批量大小（batch size）
  4. 冻结更多组件
