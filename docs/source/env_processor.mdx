# 环境处理器

环境处理器是 LeRobot 数据处理架构中的一个关键层次，负责处理**环境特定**的数据转换，与策略特定的处理相分离。这种关注点分离带来了更清晰的代码、更好的模块化，以及在不同环境和策略之间进行实验的便利性。

## 为什么需要环境处理器？

在处理不同的机器人环境（LIBERO、MetaWorld、Aloha 等）时，每个环境通常有独特的数据格式、坐标系统和约定，这些都需要在策略处理**之前**进行标准化。如果没有环境处理器，这些转换将面临以下问题：

1. **硬编码在环境代码中**——使得尝试不同的状态表示变得困难
2. **在多个策略间重复**——每个策略都需要处理环境特定的特殊问题
3. **与策略逻辑混合在一起**——违背了关注点分离原则，使调试更加困难

环境处理器通过在原始环境观测值和策略输入之间提供一个**专用的处理层**来解决这些问题。

## 处理管线

以下是在评估过程中数据如何流经完整的处理管线：

```python
# 在 lerobot_eval.py 的 rollout() 函数中：

# 1. 原始环境观测值（numpy 数组，各种格式）
raw_observation = env.step(action)

# 2. 将 numpy 转换为 torch，归一化图像 [0,1]
observation = preprocess_observation(raw_observation)

# 3. 添加任务元数据（用于多任务环境）
observation = add_envs_task(env, observation)

# 4. 环境特定的预处理（新增！）
#    - 展平机器人状态
#    - 旋转图像以匹配数据集约定
#    - 处理环境特定的坐标系
observation = env_preprocessor(observation)

# 5. 策略特定的预处理
#    - 使用数据集统计信息进行归一化
#    - 添加批次维度
#    - 移动到 GPU
#    - 对语言指令进行分词
observation = preprocessor(observation)

# 6. 策略推理
action = policy.select_action(observation)

# 7. 策略特定的后处理
#    - 反归一化动作
#    - 移除批次维度
action = postprocessor(action)

# 8. 环境特定的后处理（新增！）
#    - 必要时转换动作格式
#    - 应用环境特定的约束
action_transition = {"action": action}
action_transition = env_postprocessor(action_transition)
action = action_transition["action"]

# 9. 在环境中执行
env.step(action)
```

## 优势

### 1. **关注点分离**

环境处理器处理特定于**环境数据格式**的转换，而策略处理器处理特定于**模型需求**的转换。

```python
# ❌ 之前：关注点混杂
class LiberoVLAPolicy:
    def preprocess(self, obs):
        # 环境特定：展平机器人状态（不应该在策略中！）
        state = self._flatten_robot_state(obs["robot_state"])
        # 策略特定：使用数据集统计信息进行归一化
        state = self.normalizer(state)
        return state

# ✅ 之后：清晰分离
# 环境处理器：处理 LIBERO 的嵌套机器人状态
env_preprocessor = LiberoProcessorStep()  # 展平 robot_state

# 策略处理器：处理模型需求
policy_preprocessor = NormalizerProcessorStep(stats=dataset_stats)
```

## 使用环境处理器

### 工厂函数

`make_env_pre_post_processors` 函数遵循与策略的 `make_pre_post_processors` 相同的模式：

```python
from lerobot.envs import make_env_pre_post_processors, PushtEnv
from lerobot.envs.configs import LiberoEnv

# 对于 LIBERO：在前处理器中返回 LiberoProcessorStep
libero_cfg = LiberoEnv(task="libero_spatial", camera_name=["agentview"])
env_preprocessor, env_postprocessor = make_env_pre_post_processors(libero_cfg)

# 对于其他环境：返回恒等处理器（无操作）
pusht_cfg = PushtEnv()
env_preprocessor, env_postprocessor = make_env_pre_post_processors(pusht_cfg)
```

### 在 `envs/factory.py` 中的实现

```python
def make_env_pre_post_processors(
    env_cfg: EnvConfig,
) -> tuple[
    PolicyProcessorPipeline[dict[str, Any], dict[str, Any]],
    PolicyProcessorPipeline[dict[str, Any], dict[str, Any]],
]:
    """
    为环境观测值创建前处理器和后处理器管线。

    Args:
        env_cfg: 环境的配置。

    Returns:
        一个元组，包含：
            - preprocessor: 处理环境观测值的管线
            - postprocessor: 处理环境输出的管线
    """
    # 对于 LIBERO 环境，将 LiberoProcessorStep 添加到前处理器
    if isinstance(env_cfg, LiberoEnv) or "libero" in env_cfg.type:
        preprocessor = PolicyProcessorPipeline(steps=[LiberoProcessorStep()])
    else:
        # 对于所有其他环境，返回恒等前处理器
        preprocessor = PolicyProcessorPipeline(steps=[])

    # 后处理器当前对所有环境都是恒等的
    # 未来：可以添加环境特定的动作转换
    postprocessor = PolicyProcessorPipeline(steps=[])

    return preprocessor, postprocessor
```

### 评估中的集成

在 `lerobot_eval.py` 中，环境处理器创建一次并在整个过程中使用：

```python
def eval_main(cfg: EvalPipelineConfig):
```

## 示例：LIBERO 环境处理器

`LiberoProcessorStep` 展示了一个真实的环境处理器：

```python
from lerobot.processor import ObservationProcessorStep

@dataclass
@ProcessorStepRegistry.register(name="libero_processor")
class LiberoProcessorStep(ObservationProcessorStep):
    """
    将 LIBERO 观测值处理为 LeRobot 格式。

    **状态处理：**
    - 提取末端执行器位置（3 维）
    - 将四元数转换为轴角表示（3 维）
    - 提取夹爪关节位置（2 维）
    - 拼接为 8 维状态向量

    **图像处理：**
    - 将图像旋转 180° 以匹配 HuggingFaceVLA/libero 约定
    """

    def _process_observation(self, observation):
        processed_obs = observation.copy()

        # 处理图像：为相机约定翻转 180°
        for key in list(processed_obs.keys()):
            if key.startswith("observation.images."):
                img = processed_obs[key]
                img = torch.flip(img, dims=[2, 3])  # 翻转 H 和 W
                processed_obs[key] = img

        # 处理 robot_state：展平为 8 维向量
        if "observation.robot_state" in processed_obs:
            robot_state = processed_obs.pop("observation.robot_state")

            eef_pos = robot_state["eef"]["pos"]           # (B, 3)
            eef_quat = robot_state["eef"]["quat"]         # (B, 4)
            gripper_qpos = robot_state["gripper"]["qpos"] # (B, 2)

            # 将四元数转换为轴角
            eef_axisangle = self._quat2axisangle(eef_quat)  # (B, 3)

            # 拼接为单个状态向量
            state = torch.cat((eef_pos, eef_axisangle, gripper_qpos), dim=-1)
            state = state.float()

            processed_obs["observation.state"] = state

        return processed_obs
```

### 为什么需要这些转换？

1. **图像旋转**：HuggingFaceVLA/libero 数据集的图像相对于原始 LIBERO 模拟器旋转了 180°。处理器处理这种约定不匹配，使得在数据集上训练的策略可以无缝工作。

2. **状态展平**：原始的 LIBERO 环境暴露了包含所有可用状态信息（位置、四元数、速度、矩阵表示等）的嵌套字典。处理器会：
   - 选择相关组件（位置、四元数、夹爪）
   - 将四元数转换为轴角（更适合学习）
   - 展平为策略期望的单个 8 维向量

3. **灵活性**：环境仍然暴露**所有**原始数据。如果你想尝试不同的状态表示（例如包含速度信息、使用矩阵表示代替轴角），可以创建一个新的处理器，无需修改环境代码。

## 为新环境添加环境处理器

要为新的环境添加环境处理器：

### 1. 创建处理器步骤

```python
# 在 src/lerobot/processor/env_processor.py 中

@dataclass
@ProcessorStepRegistry.register(name="myenv_processor")
class MyEnvProcessorStep(ObservationProcessorStep):
    """处理 MyEnv 的观测值。"""

    def _process_observation(self, observation):
        processed = observation.copy()

        # 你的环境特定转换
        if "myenv.specific.state" in processed:
            state = processed.pop("myenv.specific.state")
            # 转换为标准格式
            processed["observation.state"] = self._transform_state(state)

        return processed
```

### 2. 更新你的 `EnvConfig` 子类

```python
# 在 src/lerobot/envs/factory.py 中

def make_env_pre_post_processors(env_cfg: EnvConfig):
    if isinstance(env_cfg, LiberoEnv) or "libero" in env_cfg.type:
        preprocessor = PolicyProcessorPipeline(steps=[LiberoProcessorStep()])
    elif isinstance(env_cfg, MyEnvConfig) or "myenv" in env_cfg.type:
        preprocessor = PolicyProcessorPipeline(steps=[MyEnvProcessorStep()])
    else:
        preprocessor = PolicyProcessorPipeline(steps=[])

    postprocessor = PolicyProcessorPipeline(steps=[])
    return preprocessor, postprocessor
```

### 3. 在评估中使用

无需修改！评估脚本会自动使用合适的处理器：

```bash
lerobot-eval \
    --policy.path=lerobot/my_policy \
    --env.type=myenv \  # 自动使用 MyEnvProcessorStep
    --eval.n_episodes=10
```

## 未来：环境后处理器

目前，所有环境的后处理器都是恒等的（无操作）。未来的使用场景包括：

### 动作空间转换

```python
@dataclass
class MyEnvActionPostprocessor(ProcessorStep):
    """将策略动作转换为环境特定的格式。"""

    def __call__(self, transition: EnvTransition) -> EnvTransition:
        action = transition["action"]

        # 示例：从笛卡尔空间转换为关节空间
        if self.action_space == "joint":
            action = self.ik_solver(action)

        # 示例：应用环境特定的安全限制
        action = torch.clamp(action, self.min_action, self.max_action)

        transition["action"] = action
        return transition
```

### 坐标系转换

```python
@dataclass
class CoordinateTransformPostprocessor(ProcessorStep):
    """在坐标系之间转换动作。"""

    def __call__(self, transition: EnvTransition) -> EnvTransition:
        action = transition["action"]

        # 示例：策略输出在世界坐标系，环境期望基座坐标系
        action = self.world_to_base_transform(action)

        transition["action"] = action
        return transition
```

## 最佳实践

1. **保持环境处理器简洁**：它们应仅处理环境特定的数据格式问题，而非复杂的学习相关转换。

2. **使用策略处理器处理模型需求**：归一化、批处理、设备放置和分词应属于策略处理器。

3. **从环境暴露所有数据**：让处理器决定使用哪些数据，而不是在环境中硬编码选择。

4. **记录约定**：清晰记录你的处理器处理的任何坐标系约定、相机朝向或数据格式。

5. **独立测试**：环境处理器应能在不加载完整策略或环境的情况下进行测试。

## 总结

环境处理器在环境特定的数据转换和策略特定的模型需求之间提供了**清晰的分离**。这种架构：

- ✅ 能够轻松实验不同的状态表示
- ✅ 允许策略在不同环境中无缝工作
- ✅ 保持环境代码专注于仿真/硬件接口
- ✅ 使处理器管线更易于维护和调试
- ✅ 遵循单一职责原则

关键在于：**环境定义数据格式，处理器标准化数据，策略消费标准化数据。** 每个层次都有清晰、专注的职责。
