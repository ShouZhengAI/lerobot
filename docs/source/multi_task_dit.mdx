# 多任务 DiT 策略

多任务扩散 Transformer（Diffusion Transformer，DiT）策略是原始扩散策略架构的演进，它利用带有文本和视觉条件的大型 DiT 进行多任务机器人学习。该实现同时支持用于动作生成的扩散目标和流匹配目标，使机器人能够根据语言指令执行多样化的操作任务。

## 模型概述

该模型使用：

- **CLIP 视觉编码器**：处理来自多个摄像头视角的 RGB 图像
- **CLIP 文本编码器**：编码语言任务指令（冻结权重，带有可学习的投影层）
- **扩散 Transformer**：预测以观测和语言为条件的动作序列
- **两种目标**：支持扩散（DDPM/DDIM）和流匹配两种动作生成方式

该模型的令人兴奋之处在于，您只需约 4.5 亿参数和显著更少的训练量，就能实现极高的灵巧操作能力，与数十亿参数级别的视觉语言动作（VLA）模型相媲美。

## 安装要求

多任务 DiT 策略有额外的依赖项。请通过以下命令安装：

```bash
pip install lerobot[multi_task_dit]
```

这将安装所有必要的依赖项，包括用于 CLIP 模型的 HuggingFace Transformers 库。

## 使用方法

要在 LeRobot 配置中使用多任务 DiT，将策略类型指定为：

```python
policy.type=multi_task_dit
```

## 训练

### 基本训练命令

以下是在您的数据集上训练多任务 DiT 的完整训练命令：

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/multitask_dit_training \
  --batch_size=32 \
  --steps=5000 \
  --save_freq=500 \
  --log_freq=100 \
  --policy.type=multi_task_dit \
  --policy.device=cuda \
  --policy.repo_id="HF_USER/multitask-dit-your-robot" \
  --wandb.enable=true
```

### 推荐超参数和数据集详情（30Hz 控制频率）

为获得可靠的性能，请从以下建议的默认超参数开始：

```bash
lerobot-train \
  --dataset.repo_id=YOUR_DATASET \
  --output_dir=./outputs/mutitask_dit_training \
  --batch_size=320 \
  --steps=30000 \
  --policy.type=multi_task_dit \
  --policy.device=cuda \
  --policy.horizon=32 \
  --policy.n_action_steps=24 \
  --policy.objective=diffusion \
  --policy.noise_scheduler_type=DDPM \
  --policy.num_train_timesteps=100 \
  --policy.repo_id="HF_USER/multitask-dit-your-robot" \
  --wandb.enable=true
```

**关键参数：**

- **批量大小**：192-320 - 如果您有支持此批量的 GPU，将获得最佳训练效果
- **Horizon（预测范围）**：32 - 预测的动作步数，30Hz 下约 1.0 秒
- **n_action_steps（动作步数）**：24 - 30Hz 下约 0.8 秒
- **Objective（目标）**：`diffusion` - 从扩散开始，如果生成质量不佳，可以尝试流匹配
- **训练步数**：单任务建议超过 3 万步

### 训练配置参数

#### 目标选择

在扩散和流匹配之间选择：

```bash
# 扩散目标（默认）
--policy.objective=diffusion \
--policy.noise_scheduler_type=DDPM \  # 或 "DDIM"
--policy.num_train_timesteps=100 \
--policy.num_inference_steps=10 \  # 用于更快的推理
--policy.beta_schedule=squaredcos_cap_v2 \  # 噪声调度类型
--policy.prediction_type=epsilon \  # "epsilon"（预测噪声）或 "sample"（预测干净信号）
--policy.clip_sample=true \  # 去噪过程中裁剪样本
--policy.clip_sample_range=1.0  # 裁剪范围 [-x, x]

# 流匹配目标
--policy.objective=flow_matching \
--policy.timestep_sampling_strategy=beta \  # 或 "uniform" | beta 采样策略在实践中效果明显更好
--policy.num_integration_steps=100 \
--policy.integration_method=euler \  # 或 "rk4"
--policy.sigma_min=0.0  # 流插值路径中的最小噪声
```

#### Transformer 架构

根据数据集大小调整模型容量：

```bash
# 小型数据集（少于 100 个样本）
--policy.num_layers=4 \
--policy.hidden_dim=512 \
--policy.num_heads=8  # 理想情况下应为 hidden_dim // 64

# 中型数据集（100-5000 个样本）- 默认
--policy.num_layers=6 \
--policy.hidden_dim=512 \
--policy.num_heads=8  # 理想情况下应为 hidden_dim // 64

# 大型数据集（超过 5000 个样本）
--policy.num_layers=8 \
--policy.hidden_dim=512 \
--policy.num_heads=8   # 理想情况下应为 hidden_dim // 64
```

**位置编码选项：**

模型支持两种动作序列位置编码方法：

```bash
# 旋转位置编码（RoPE）- 默认，推荐
--policy.use_rope=true \
--policy.rope_base=10000.0  # RoPE 的基础频率

# 绝对位置编码
--policy.use_positional_encoding=true  # 启用时禁用 RoPE
```

**其他 Transformer 参数：**

```bash
--policy.dropout=0.1  # DiT 块的 Dropout 比率（0.0-1.0）
--policy.timestep_embed_dim=256  # 时间步嵌入维度
```

#### 视觉编码器配置

```bash
# 使用不同的 CLIP 模型以获得更强的表现力，但推理时间会增加
# 根据任务的复杂度和数据集大小尝试更大或更小的模型
--policy.vision_encoder_name=openai/clip-vit-large-patch14

# 每个摄像头使用独立的视觉编码器
# 当摄像头具有显著不同的特性时可能有用，但要注意显存占用增加
--policy.use_separate_rgb_encoder_per_camera=true

# 图像预处理
--policy.image_resize_shape=[XXX,YYY] \ # 您可能需要调整图像大小以加快推理速度
--policy.image_crop_shape=[224,224] \
--policy.image_crop_is_random=true  # 训练时随机裁剪，推理时中心裁剪
```

#### 文本编码器配置

```bash
# 使用不同的 CLIP 文本编码器模型
# 与视觉编码器类似：根据任务复杂度和数据集大小尝试更大或更小的模型
--policy.text_encoder_name=openai/clip-vit-large-patch14
```

#### 学习率配置

视觉编码器使用独立的学习率乘数，建议从 1/10 开始：

```bash
--policy.optimizer_lr=2e-5 \
--policy.vision_encoder_lr_multiplier=0.1  # 视觉编码器学习率 = 0.1 * optimizer_lr
```

### 训练调优指南

#### 1. 带 Beta 采样的流匹配

此处的原始扩散实现基于 [TRI 的 LBM 论文](https://arxiv.org/abs/2507.05331) 中描述的工作。

此外，我们还实现了流匹配目标，在 [Boston Dynamics 博客文章](https://bostondynamics.com/blog/large-behavior-models-atlas-find-new-footing/) 中有高级别描述。

请考虑测试流匹配目标并评估您的任务在性能上的差异：

```bash
--policy.objective=flow_matching \
--policy.timestep_sampling_strategy=beta \
--policy.timestep_sampling_alpha=1.5 \
--policy.timestep_sampling_beta=1.0 \
--policy.timestep_sampling_s=0.999
```

这并非在所有使用场景下都是万能的，但有时会带来更平滑、更一致的动作。

#### 2. Transformer 层数

根据数据集大小匹配模型容量：

- **小型数据集**（少于 100 个样本）：减少到 4 层
- **大型数据集**（超过 5000 个样本）：增加到 8 层

#### 3. `horizon`（预测范围）调优

模型对您选择的预测范围可能比较敏感。从大约 1 秒的范围开始，根据您的控制频率调整：

- **30 Hz 频率**：`horizon=30`
- **10 Hz 频率**：`horizon=10`

然后从此逐步增加尝试。horizon 决定了模型预测动作的未来范围。

#### 4. `n_action_steps`（动作步数）敏感性

模型对 `n_action_steps` 也可能非常敏感。从大约 0.8 秒开始，根据您的控制频率调整：

- **较小的值**：反应更快，但可能对长期任务不够稳定
- **较大的值**：更适合长期执行，但开环失败后的恢复能力有限

### 推理调优

对于更快的推理，使用 DDIM 和更少的采样步数：

```bash
--policy.noise_scheduler_type=DDIM \
--policy.num_inference_steps=10
```

### 恢复训练

要从检查点恢复训练：

```bash
lerobot-train \
  --config_path=./outputs/mutitask_dit_training/checkpoints/last/pretrained_model/train_config.json \
  --resume=true
```

检查点目录应包含 `model.safetensors` 和 `config.json` 文件（训练过程中自动保存）。恢复时，配置从检查点加载，因此您无需指定其他参数。

## 常见故障模式与调试

训练这些模型可能比较棘手。以下是常见的故障模式和调试方法：

### 静止不动/无动作

模型可能在推理时"崩溃"，导致静止或没有动作。这可能在以下情况下发生：

1. **训练数据不足**：如果您只有 20-50 个样本，请尝试将数据集大小大致翻倍。一旦超过 300 个样本，如果仍然出现此问题，可能是任务过于复杂。

2. **多个相似任务**：当您的数据集包含多个相似任务时（例如拿起 2 个不同的物体），模型可能过度依赖语言条件，而这些条件可能不够丰富。

**调试建议：**

- 增加数据集大小（翻倍直到超过 300 个样本）
- 延长训练时间，最多到 10 万步，即使损失已经持平
- 检查模型是否接收到正确的语言指令，或增加指令的多样性

### 执行错误的任务

有时机器人会完全忽视您的指令而执行其他任务。这通常只在您训练了多个任务时才会发生。

**可能的原因：**

- 语言指令歧义
- 特定任务的训练数据不足
- 模型对多任务数据集中相似任务之间的混淆

**调试建议：**

- 验证语言指令的特异性，尤其是多个任务的描述相似时
- 检查训练数据集中的任务分布，并对失败/被忽略的任务增加权重
- 考虑进行特定任务的微调

### 训练不稳定

如果训练损失不稳定或发散：

- 尝试在 `1e-5` 到 `3e-4` 之间调整学习率
- 如果可能，增大批量大小
- 检查数据集归一化是否正确
- 验证图像预处理是否正确运行

## 性能考虑

### GPU 要求

- **推理**：建议至少使用 RTX 5070 Ti（或同等 GPU）以获得合理的速度性能
- **训练**：具有足够显存来加载大于 64 的批量大小的 GPU 是理想的，具体取决于图像观测数量等

### 批量大小推荐

- **最低要求**：64（低于此值可能导致训练不稳定）
- **推荐值**：256-320（性能最佳，需要更大的 GPU）
