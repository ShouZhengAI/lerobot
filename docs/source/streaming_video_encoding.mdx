# 流式视频编码指南

## 1. 概述

流式视频编码（Streaming video encoding）消除了视频数据集录制过程中传统的 PNG 中间步骤。不再需要：

1. 捕获帧 -> 将 PNG 写入磁盘 ->（片段结束后）读取 PNG -> 编码为 MP4 -> 删除 PNG

现在帧可以在捕获过程中实时编码：

1. 捕获帧 -> 加入编码器线程队列 -> 直接编码为 MP4

这使得 `save_episode()` 近乎瞬时完成（片段结束时视频已经编码完毕），并消除了之前片段之间可能出现的阻塞等待，特别是在多摄像头长片段录制时。

## 2. 调优参数

| 参数                     | CLI 参数                            | 类型            | 默认值         | 说明                                                   |
| ------------------------ | ----------------------------------- | --------------- | -------------- | ------------------------------------------------------ |
| `streaming_encoding`     | `--dataset.streaming_encoding`      | `bool`          | `True`         | 在捕获过程中启用实时编码                                |
| `vcodec`                 | `--dataset.rgb_encoder.vcodec`      | `str`           | `"libsvtav1"`  | 视频编码器。`"auto"` 自动检测最佳硬件编码器             |
| `encoder_threads`        | `--dataset.encoder_threads`         | `int \| None`   | `None`（自动） | 每个编码器实例使用的线程数。`None` 让编码器自行决定     |
| `encoder_queue_maxsize`  | `--dataset.encoder_queue_maxsize`   | `int`           | `30`           | 每个摄像头最大缓冲帧数（30fps 约 1 秒）。会消耗内存     |

## 3. 性能考量

流式编码意味着 CPU 在**捕获循环期间**就对视频进行编码，而非事后。这产生了一个需要在以下各方之间共享的 CPU 预算：

- **控制循环**（读取摄像头、控制机器人、写入非视频数据）
- **编码器线程**（每个摄像头一个线程池）
- **Rerun 可视化**（如果启用）
- **操作系统及其他进程**

### 分辨率与摄像头数量影响

| 配置                         | 吞吐量（像素/秒） | CPU 编码负载 | 说明                           |
| ---------------------------- | ----------------- | ------------ | ------------------------------ |
| 2 个摄像头 640x480x3 @30fps  | 55M               | 低           | 在大多数系统上均可运行         |
| 2 个摄像头 1280x720x3 @30fps | 165M              | 中等         | 现代系统流畅运行               |
| 2 个摄像头 1920x1080x3 @30fps| 373M              | 高           | 需要强大的高端 CPU             |

### `encoder_threads` 调优

此参数控制每个编码器实例内部使用的线程数：

- **较大值**（如 4-5）：编码更快，但每个摄像头占用更多 CPU 核心。适用于多核高端系统。
- **较小值**（如 1-2）：每个摄像头占用较少 CPU，释放核心给捕获和可视化使用。适用于低分辨率图像和性能较强的 CPU。
- **`None`（默认值）**：让编码器自行决定。可在编码器日志中查看具体信息。

### 背压与丢帧

每个摄像头都有一个有界队列（`encoder_queue_maxsize`，默认 30 帧）。当编码器跟不上时：

1. 队列被填满（消耗内存）
2. 新帧会被**丢弃**（不会阻塞）——捕获循环继续运行不受影响
3. 记录警告日志：`"Encoder queue full for {camera}, dropped N frame(s)"`
4. 片段结束时，报告每个摄像头丢弃的总帧数

### 编码器跟不上的症状

- **系统感觉卡顿和冻结**：所有 CPU 占用率达到 100%
- **日志中出现丢帧警告**，或录制数据集的帧数/FPS 低于预期
- **机器人运动卡顿**：如果 CPU 严重过载，捕获循环本身也可能受到影响
- **Rerun 延迟累积**：可视化落后于实时

## 4. 硬件加速编码

### 何时使用

在以下情况下使用硬件编码：

- CPU 是瓶颈（丢帧、机器人卡顿、Rerun 延迟）
- 你有兼容的硬件（GPU 或专用编码器）
- 你以高吞吐量录制（高分辨率或多摄像头）

### 选择编码器

| 编码器                  | CPU 使用率 | 文件大小      | 质量   | 说明                                                             |
| ----------------------- | ---------- | ------------- | ------ | ---------------------------------------------------------------- |
| `libsvtav1`（默认）     | 高         | 最小          | 最佳   | 默认值。压缩率最高但最耗 CPU                                     |
| `h264`                  | 中         | 大约大 30-50% | 好     | 软件 H.264。CPU 占用较低                                         |
| 硬件编码器              | 非常低     | 最大          | 好     | 卸载到专用硬件。最适合 CPU 受限的系统                            |

### 可用的硬件编码器

| 编码器               | 平台          | 硬件                                                                                           | CLI 值                                               |
| -------------------- | ------------- | ---------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| `h264_videotoolbox`  | macOS         | Apple Silicon / Intel                                                                          | `--dataset.rgb_encoder.vcodec=h264_videotoolbox`     |
| `hevc_videotoolbox`  | macOS         | Apple Silicon / Intel                                                                          | `--dataset.rgb_encoder.vcodec=hevc_videotoolbox`     |
| `h264_nvenc`         | Linux/Windows | NVIDIA GPU                                                                                     | `--dataset.rgb_encoder.vcodec=h264_nvenc`            |
| `hevc_nvenc`         | Linux/Windows | NVIDIA GPU                                                                                     | `--dataset.rgb_encoder.vcodec=hevc_nvenc`            |
| `h264_vaapi`         | Linux         | Intel/AMD GPU                                                                                  | `--dataset.rgb_encoder.vcodec=h264_vaapi`            |
| `h264_qsv`           | Linux/Windows | Intel Quick Sync                                                                               | `--dataset.rgb_encoder.vcodec=h264_qsv`              |
| `auto`               | 任意          | 探测系统可用的硬件编码器。若未找到硬件编码器则回退到 `libsvtav1`                               | `--dataset.rgb_encoder.vcodec=auto`                  |

> [!NOTE]
> 要使用硬件加速编码器，你可能需要升级 GPU 驱动程序。

> [!NOTE]
> `libsvtav1` 是默认编码器，因为它提供了最佳的训练性能；其他编码器可以减少 CPU 使用率并提高速度，但通常会产生更大的文件，并可能影响训练时间。

## 5. 故障排除

| 症状                                                             | 可能原因                                        | 解决方案                                                                                                                                                                                                                                                                                           |
| ---------------------------------------------------------------- | ----------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 系统卡死、机器人运动卡顿或 Rerun 可视化延迟                      | CPU 资源不足（100% 负载）                       | 关闭其他应用程序，降低编码吞吐量，降低 `encoder_threads`，使用 `h264`，使用 `display_data=False`。如果 CPU 仍保持在 100%，则可能无法满足你的配置需求，考虑使用 `--dataset.streaming_encoding=false` 或硬件编码 `--dataset.rgb_encoder.vcodec=auto`                                                  |
| "Encoder queue full" 警告或数据集中出现丢帧                       | 编码器跟不上（队列溢出）                         | 如果 CPU 未达 100%：增加 `encoder_threads`，增加 `encoder_queue_maxsize`，或使用硬件编码 `--dataset.rgb_encoder.vcodec=auto`                                                                                                                                                                       |
| 内存占用高                                                        | 队列填充速度快于编码速度                         | `encoder_threads` 太低或 CPU 不足。减少 `encoder_queue_maxsize` 或使用硬件编码                                                                                                                                                                                                                     |
| 视频文件过大                                                      | 使用了硬件编码器或 H.264                        | 这是预期的权衡。如果 CPU 允许，切换到 `libsvtav1`                                                                                                                                                                                                                                                  |
| `save_episode()` 仍然很慢                                        | `streaming_encoding` 为 `False`                 | 设置 `--dataset.streaming_encoding=true`                                                                                                                                                                                                                                                           |
| 编码器线程崩溃                                                    | 编码器不可用或设置无效                           | 检查 `vcodec` 是否已安装，尝试 `--dataset.rgb_encoder.vcodec=auto`                                                                                                                                                                                                                                |
| 录制的数据集缺少帧                                                | CPU/GPU 资源不足或偶发性负载激增                 | 如果缺失约 5% 的帧，说明系统可能过载——请遵循上述建议。如果缺失较少（约 2%），很可能是偶发性的瞬时负载激增（常在启动时出现），属于预期情况。                                                                                                                          |

## 6. 推荐配置

以下估算偏保守；建议在你的硬件上测试——从较低负载开始，逐步增加。

### 高端系统：现代 12 核以上（24 线程以上）

CPU 模式下吞吐量在约 250-500M 像素/秒范围内应可流畅运行。如果可用，尝试硬件编码可获得更好的效果。

```bash
# 3 个摄像头 1280x720x3 @30fps：默认设置即可良好运行。可选增加编码器并行度。
# 2 个摄像头 1920x1080x3 @30fps：默认设置即可良好运行。可选增加编码器并行度。
lerobot-record --dataset.encoder_threads=5 ...

# 3 个摄像头 1920x1080x3 @30fps：可能需要一些调优。
```

### 中端系统：现代 8 核以上（16 线程以上）或 Apple Silicon

CPU 模式下吞吐量在约 80-300M 像素/秒范围内应可实现。

```bash
# 3 个摄像头 640x480x3 @30fps：默认设置即可良好运行。可选降低编码器并行度。
# 2 个摄像头 1280x720x3 @30fps：默认设置即可良好运行。可选降低编码器并行度。
lerobot-record --dataset.encoder_threads=2 ...

# 2 个摄像头 1920x1080x3 @30fps：可能需要一些调优。
```

### 低资源系统：现代 4 核以上（8 线程以上）或 Raspberry Pi 5

在非常受限的系统上，流式编码可能与捕获循环争抢过多的 CPU 资源。禁用它回退到基于 PNG 的方式，编码在片段之间进行（阻塞，但不会干扰捕获）。或者，以更低的吞吐量录制以减少捕获和编码负载。也可以考虑将编码器切换为 `h264` 并使用批量编码。

```bash
# 2 个摄像头 640x480x3 @30fps：需要一些调优。

# 使用 H.264，禁用流式编码，考虑批量编码
lerobot-record --dataset.rgb_encoder.vcodec=h264 --dataset.streaming_encoding=false ...
```

## 7. 结语

性能最终取决于你的具体配置——每秒帧数、分辨率、CPU 核心与负载、可用内存、片段时长以及你选择的编码器。务必使用你的目标工作负载进行测试，注意你的 CPU 和系统能力，并合理调优 `encoder_threads`、`encoder_queue_maxsize` 和 `vcodec`。也就是说，一个常见的实用配置（适用于许多应用）是三个 640x480x3 @30fps 的摄像头；这在现代系统上使用默认的流式视频编码设置通常可以良好运行。请始终通过比较视频时长与 CLI 片段时长，并确认行数等于 FPS 乘以 CLI 时长，来验证录制的数据集是否健康。
