# Isaac Teleop

使用 NVIDIA [Isaac Teleop](https://github.com/NVIDIA/IsaacTeleop) 控制你的机器人，这是一个多模态遥操作框架。Isaac Teleop 通过单个 `TeleopSession` 驱动多种输入设备——XR（VR）控制器、手部追踪、全身追踪、Manus 手套、脚踏板等。

在 LeRobot 中，Isaac Teleop 作为一个独立的示例，位于
[`examples/isaac_teleop_to_so101/`](https://github.com/huggingface/lerobot/tree/main/examples/isaac_teleop_to_so101)。
每个 Isaac Teleop 输入设备都是示例 `isaac_teleop` 包中的一个 `Teleoperator` 子类，共享同一个会话生命周期（参见 `IsaacTeleopTeleoperator`）。目前可用的设备有 **XR 控制器**（`XRController`）和可反驱的 **SO-101 主端（leader）机械臂**（`SO101LeaderArm`）；Manus 手套以及手部/全身追踪是下一步自然可添加的设备。本指南重点介绍 XR 控制器；SO-101 主端的摘要见[运行示例](#step-3-run-the-example)部分。

**在本指南中你将学习：**

- Isaac Teleop 设备如何驱动机器人末端执行器（EE）目标
- 离合器（clutch）（XR 控制器上的握持/抓握）如何在避免机械臂跳动的情况下接合遥操作
- 如何运行 SO-101 遥操作示例并调节运动/夹爪/逆运动学（IK）

## 安装

该示例位于 LeRobot 仓库中（不属于 `lerobot` pip 包），因此需要克隆仓库并从源码安装。规范且始终最新的安装和使用参考是该示例的
[`README.md`](https://github.com/huggingface/lerobot/tree/main/examples/isaac_teleop_to_so101/README.md)；
简要来说：

```bash
git clone https://github.com/huggingface/lerobot.git
cd lerobot
uv pip install -e ".[feetech,kinematics,dataset]" "huggingface_hub>=1.5"
uv pip install "isaacteleop[cloudxr,retargeters-lite]~=1.3.131" "scipy>=1.14"
```

`isaacteleop` 发布在公共 PyPI 上（仅 Linux）。`cloudxr` 附加组件带来 CloudXR 运行时绑定；`retargeters-lite` 是基于 scipy 的重定向器（retargeter）路径，可在 x86_64 和 ARM 上解析（在 aarch64 上——例如 DGX Spark——完整的 `retargeters` 附加组件因 `dex-retargeting`/`nlopt` 的固定版本要求而无法解析，这就是此处不将其设为默认的原因）。在 x86_64 上，你还可以额外安装完整的重定向器栈：

```bash
uv pip install "isaacteleop[retargeters]~=1.3.131"
```

### 设置 CloudXR 并连接头戴设备（headset）

Isaac Teleop 通过 **NVIDIA CloudXR** 将头戴设备串流到你的机器，CloudXR 提供了会话连接的 OpenXR 运行时。默认情况下，LeTeleop **在调用 `teleop_device.connect()` 时自动为你启动 CloudXR 运行时**——你不再需要在单独的 shell 中运行 `python -m isaacteleop.cloudxr` 和 `source cloudxr.env`。你只需要连接一个受支持的头戴设备并开放 CloudXR 防火墙端口。请遵循 Isaac Teleop
[快速入门](https://nvidia.github.io/IsaacTeleop/main/getting_started/quick_start.html)了解头戴设备配对和防火墙的详细信息。

**首次运行（EULA）。** 首次启动必须接受 NVIDIA CloudXR EULA。自动启动会在 **stdin** 上提示接受 EULA，因此在无头机器上它会挂起等待输入。请以交互方式一次性引导 EULA：

```bash
python -m isaacteleop.cloudxr --accept-eula   # 一次性操作：接受 CloudXR EULA
```

之后，`connect()` 将以非交互方式启动运行时。启动过程**阻塞约 30 秒**，直到运行时启动完成。

**配置。** `IsaacTeleopConfig` 上的两个字段（所有设备共享）控制此行为：

- `auto_launch_cloudxr`（默认 `True`）—— 是否在 `connect()` 时启动运行时。当 CloudXR 已在外部运行时，设置为 `False`。
- `cloudxr_env_file`（默认 `None`）—— 可选的 CloudXR 设备配置文件 `.env`，用于选择头戴设备传输方式（例如 Apple Vision Pro 配置文件）。这是启动器的**输入**；它不是旧手动流程中告诉你要 `source` 的 `~/.cloudxr/run/cloudxr.env` **输出**文件。`None` 保留默认的自动 WebRTC 配置文件——不过 SO-101 示例将其覆盖为由 `teleoperate.py` 旁边的 `default.env` 提供，除非你传递 `--teleop.cloudxr_env_file`。

**选择退出。** 要跳过自动启动（CloudXR 已在运行），可以设置 `auto_launch_cloudxr=False` 或导出：

```bash
export LEROBOT_CLOUDXR_SKIP_AUTOLAUNCH=1
```

**环境变量优先于配置字段**：如果设置了 `LEROBOT_CLOUDXR_SKIP_AUTOLAUNCH=1`，即使 `auto_launch_cloudxr=True` 也会跳过自动启动。此变量**独立于** Isaac Lab 的 `ISAACLAB_CXR_SKIP_AUTOLAUNCH`——设置一个不会影响另一个。

**每个进程一个遥操作器。** CloudXR 运行时在进程范围内配置环境（单例），因此每个进程运行一个 Isaac Teleop 遥操作器。

**关闭。** 退出时务必调用 `teleop_device.disconnect()`——包括在 Ctrl-C 时。将你的遥操作循环包装在 `try/finally` 中，并在 `finally` 中调用 `disconnect()`。这会在 CloudXR 运行时**之前**拆除 OpenXR 会话，这是必需的顺序；启动器的 `atexit` 钩子只回收运行时，不执行会话的 `__exit__`，因此如果没有显式的 `disconnect()`，被中断的运行会以错误的顺序关闭。

```python
teleop_device.connect()
try:
    while True:
        action = teleop_device.get_action()
        # ... 驱动机器人 ...
finally:
    teleop_device.disconnect()
```

参见[系统要求](https://nvidia.github.io/IsaacTeleop/main/references/requirements.html)了解支持的 OS / GPU / CloudXR 版本和头戴设备。

## 工作原理

XR 控制器是一种 Isaac Teleop **输入**设备。`XRController` 是一个刻意精简的读取器：它暴露 **原始** 控制器抓握位姿——已经静态重基到机器人基座坐标系——加上握持（squeeze）和扳机（trigger）的模拟值。它**没有**重定向器，也没有自己的离合器逻辑。离合器（接合锁存 + 基于 EE 的增量重基）和夹爪映射位于示例循环的下游，然后输入到 LeRobot 现有的闭环笛卡尔 IK 流程——与手机遥操作器使用的流程相同。设备特定的部分有 `XRController`、循环中的 `Clutch` 和 `MapXRControllerActionToRobotAction`；下游的所有内容（`EEBoundsAndSafety`、`InverseKinematicsEEToJoints`）是共享的，未来的设备（例如 Manus 手套）会替换为自己的 `teleop_<device>.py` 和处理器，同时复用其余部分。

`XRController._build_pipeline` 连接 Isaac Teleop 的 `ControllersSource`——已由原生的 `ControllerTransform`（`base_T_anchor`）静态重基到机器人基座坐标系——并逐字公开转换后的控制器流。`get_action()` 直接读取抓握位姿、握持值和扳机值；会话始终以 `RUNNING` 状态步进（没有离合器重定向器来门控）。

`Clutch` 类（位于 `examples/isaac_teleop_to_so101/isaac_teleop/clutch.py`，由 `common.py` 中的循环驱动）镜像了 Isaac Teleop 的 `SO101ClutchRetargeter`，但驻留在循环中，以便设备保持精简读取器的角色：

- 它在握持**接合边沿**（握持值首次超过 `clutch_threshold` 的时刻）锁存其接合原点，并围绕该原点重基位置和方向，以便接合时不会导致机械臂瞬移。`Clutch.rebase` 返回绝对基座坐标系目标（`pos`、`quat`）对，循环将其拼接成馈送到处理器的 7 维 `ee_pose`。
- 模拟扳机变成 `[0, 1]` 范围内的夹爪关闭度 `closedness`（0 = 打开，1 = 关闭），与扳机拉动成比例，`MapXRControllerActionToRobotAction` 将其映射到夹爪目标。

参见 Isaac Teleop
[重定向接口](https://nvidia.github.io/IsaacTeleop/main/references/retargeting/index.html)
和[架构概述](https://nvidia.github.io/IsaacTeleop/main/overview/architecture.html)
了解源节点和重定向器如何组合。

```text
  VR 控制器（OpenXR）
        │
        ▼
  XRController.get_action()          ── 原始基座坐标系 grip_pos / grip_quat + squeeze + trigger
        │                                （TeleopSession 始终以 RUNNING 步进；离合器在流程下游）
        ▼
  Clutch.rebase(grip_pos, grip_quat) ── 应用于 EE home 的接合相对增量（位置 + 方向）
        │  ee_pose (7) / closedness       → 绝对 ee_pose；closedness = trigger
        ▼
  MapXRControllerActionToRobotAction ── 绝对 ee.x/y/z；ee.w* = 方向 rotvec 目标；
        │  ee.x/y/z / ee.w* / ee.gripper_pos   ee.gripper_pos = (1 - closedness) * 100
        ▼
  EEBoundsAndSafety                ── 工作空间裁剪 + 每帧步长限制（截断+警告）
        │
        ▼
  InverseKinematicsEEToJoints      ── 闭环 Placo IK；位置 + 软方向
        │  （orientation_weight=0.01）   （传递 ee.gripper_pos → gripper.pos）
        ▼
  SO-101 从端关节目标
```

### 离合器：由示例循环拥有

与手机流程（将离合器分散在 `MapPhoneActionToRobotAction` 和 `EEReferenceAndDelta` 之间）不同，XR 离合器完全存在于示例循环的 `Clutch` 类中。它发出一个**绝对**末端执行器位姿，因此没有 `EEReferenceAndDelta` 阶段，处理器中也没有增量累积——`MapXRControllerActionToRobotAction` 是一个纯粹的无状态逐帧映射。

离合器在握持**接合边沿**（握持值超过 `clutch_threshold` 的时刻）锁存其接合原点，并从相对于该原点的运动来驱动 EE，因此机械臂在接合时不会瞬移。在**每次**接合时——无论是启动还是任务中间的重新离合——home _位置_ 都是从机械臂的**测量关节值**通过正向运动学锁存，因此即使机械臂在脱离状态下移动过，home 也等于机械臂实际所在的位置，接合是无跳变的。home _方向_ 则保持上一次指令的旋转：5 自由度机械臂只是软性跟踪方向，因此在每次重新离合时将测量的手腕方向锁存会将其跟踪偏移注入指令。

## 控制方式

- **握持 / 抓握** —— **离合器**（死人手柄）。按住超过 `clutch_threshold` 可接合遥操作；松开则暂停。每次接合都会重新捕获原点，因此你可以在暂停时重新调整手的位置，然后重新接合，机械臂不会跳动（索引/离合器风格）。
- **扳机** —— **夹爪**，以**模拟量**控制。夹爪开度与扳机成比例——半按扳机使夹爪半闭——通过 `[0, 1]` 范围内的关闭度（0 = 打开，1 = 关闭）映射到绝对夹爪关节目标。
- **控制器方向** —— **手腕**。离合器将控制器方向（接合相对、基座坐标系）重基到软 IK 方向目标，手腕与位置一起跟踪。在 5 自由度 SO-101 上，手腕只是部分跟随手部——详见下面的 `orientation_weight`。

## 开始使用

### 第一步：创建遥操作器

```python
# 从仓库根目录运行，以便 `examples` 包可被导入。
from examples.isaac_teleop_to_so101.isaac_teleop import XRController, XRControllerConfig

teleop_config = XRControllerConfig(
    hand_side="right",      # "left" 或 "right" 控制器
    clutch_threshold=0.5,   # 握持值超过此阈值时离合器接合
)
teleop_device = XRController(teleop_config)
```

`XRController.get_action()` 返回**原始**基座坐标系控制器位姿，而非经过离合器重基后的目标：`grip_pos` (3,) `[x, y, z]` [米] 和 `grip_quat` (4,) `[qx, qy, qz, qw]` 在机器人基座坐标系中，以及标量 `squeeze` 和 `trigger` 模拟值，范围 `[0, 1]`。示例循环中的 `Clutch` 将这些转换为绝对 `ee_pose`，握持值由循环根据 `clutch_threshold` 进行阈值判断以接合。

### 第二步：连接

调用 `teleop_device.connect()` 首先自动启动 CloudXR 运行时（除非你选择退出——参见[设置 CloudXR 并连接头戴设备](#set-up-cloudxr-and-connect-a-headset)；这阻塞约 30 秒，首次运行时会在 stdin 上提示接受 EULA），然后启动 Isaac Teleop
[`TeleopSession`](https://nvidia.github.io/IsaacTeleop/main/getting_started/teleop_session.html)
（打开 OpenXR 会话并发现控制器）。XR 控制器是自标定的，因此没有手动标定步骤——离合器处理每次接合时的重新居中。将 `connect()` 与调用 `disconnect()` 的 `try/finally` 配对使用，以便在退出/Ctrl-C 时会话在运行时之前拆除。

### 第三步：运行示例

该示例假设你已经配置好了机器人（SO-101 从端）并设置了正确的串口。

**机器人 URDF 及其网格文件在首次运行时自动获取**：XR 设备从
[`lerobot/robot-urdfs` Hugging Face bucket](https://huggingface.co/buckets/lerobot/robot-urdfs/tree/so101)
将 SO-101 URDF 下载到 LeRobot 缓存（`HF_LEROBOT_HOME/robot-urdfs/so101/`）中，之后重复使用，因此无需单独的下载步骤：

```bash
python -m examples.isaac_teleop_to_so101.teleoperate --robot.type=so101_follower --robot.port=/dev/ttyACM0 \
    --robot.id=so101_follower_arm --teleop.type=xr_controller
```

CLI 是 `lerobot-teleoperate` 风格（draccus）：`--robot.*` 配置 SO-101 从端，`--teleop.type` 选择 Isaac 输入设备（`xr_controller` | `so101_leader`），`--teleop.*` 是其设备参数。`--teleop.type=xr_controller` 运行上述的 XR 控制器路径。启动安全约定：默认情况下，它会在 `--reset_duration` 秒内将所有关节缓动到默认复位姿势（`--reset_to_origin=false` 保持机械臂当前位置），然后从机械臂测量的位姿中播种离合器 home，使得首次接合无跳变；只有在离合器接合时才命令从端。

**自定义复位姿势。** 复位姿势作为内置默认值提供（一个舒适的中程姿势），开箱即用——你**无需**记录任何内容。要针对你的设置进行定制，将机械臂反驱到所需姿势并运行
`python -m examples.isaac_teleop_to_so101.override_reset_pose --id <robot.id>`；它会将当前关节值写入 LeRobot 缓存中的一个按手臂区分的文件
（`HF_LEROBOT_HOME/reset_poses/<robot.name>/<robot.id>.json`，键名类似标定文件），该文件在下一次运行时优先于内置默认值。由于它位于用户本地缓存中（而非仓库中），你的覆盖设置保留在你的机器上，`teleoperate` 和 `record` 在使用相同的 `--robot.id` 启动时都会遵循它。

另一个设备 `--teleop.type=so101_leader`，通过 Isaac Teleop 原生的 `so101_leader` 插件流式传输可反驱的 SO-101 **主端（leader）机械臂**的关节值，与从端 1:1 镜像（无离合器、无 IK——主端和从端共享 SO-101 运动学）。

`so101_leader_plugin` 二进制文件是一个 C++ 插件，**不属于** `isaacteleop` pip 包——你需要从 Isaac Teleop 源码树构建它。请遵循
[从源码构建 Isaac Teleop](https://nvidia.github.io/IsaacTeleop/main/getting_started/build_from_source/index.html)
（简而言之，从你的 Isaac Teleop 检出目录中：`cmake -B build && cmake --build build --parallel && cmake --install build`）；构建会将插件安装到 `<IsaacTeleop>/install/plugins/` 下，因此二进制文件位于 `install/plugins/so101_leader/so101_leader_plugin`——参见下面的 `--launch_plugin` 路径。有关串口/标定详情，请参阅插件自身的 `README.md`（与二进制文件同级）。

将 `--teleop.port` 指向物理主端的串口，将 `--launch_plugin` 指向该插件二进制文件，脚本将在 CloudXR 启动后启动它：

```bash
python -m examples.isaac_teleop_to_so101.teleoperate --robot.type=so101_follower --robot.port=/dev/ttyACM0 \
    --robot.id=so101_follower_arm --teleop.type=so101_leader \
    --teleop.port=/dev/ttyACM1 --teleop.id=so101_leader_arm \
    --launch_plugin=/code/Teleop/install/plugins/so101_leader/so101_leader_plugin
```

（注意这里的 `so101_leader` 是 _Isaac_ 主端，根据 Isaac Teleop 设备注册表解析，与 `lerobot-teleoperate` 的串口 `so101_leader` 不同。）当设置了 `--teleop.port` 时，插件的 tick→弧度标定从 `--teleop.id` 推断并作为第三个位置参数传递给插件——即 `HF_LEROBOT_CALIBRATION/teleoperators/so_leader/<id>.json` 中的 LeRobot 格式 JSON，与串口 SO-101 主端使用的文件相同（`lerobot-calibrate --teleop.type=so101_leader --teleop.id=<id>`）。如果文件缺失，脚本会发出警告，插件使用内置默认值。运行 `python -m examples.isaac_teleop_to_so101.teleoperate --help` 查看所有标志。其启动安全约定：默认情况下，从端会在 `--align_duration` 秒内缓动到主端的首次读数值（`--align=false` 可跳过），这样机械臂在镜像开始时不会突然跳动，而当主端流数据过时时，从端会保持其测量的位姿。

URDF 获取使用 `huggingface_hub`（已经是 LeRobot 的依赖项）访问公有的 `lerobot/robot-urdfs` bucket，因此无需登录。它缓存在 `HF_LEROBOT_HOME/robot-urdfs/so101/` 下；删除该文件夹可强制重新下载。

然后，在你的头戴设备中：握持并按住抓握键来接合，移动控制器以驱动机械臂，扭转/倾斜以定向手腕，按下扳机关闭夹爪（成比例——松开以打开）。

要录制数据集（而不仅仅是遥操作），请使用同一文件夹中的 `record.py`。它像 `teleoperate.py` 一样根据 `--teleop.type`（`xr_controller` | `so101_leader`）进行分发，因此任一设备都可以驱动从端，并将指令关节值保存到 LeRobot 数据集中（`lerobot-record` 风格的 `--dataset.*` 标志）。参见其模块文档字符串了解完整的 CLI 和键盘录制快捷键。

## 重要的流程步骤和选项

离合器已经产生一个绝对基座坐标系位姿，因此处理器端是一个精简的**绝对位姿**路径——没有帧重映射、没有增量累积、也没有 `EEReferenceAndDelta` 阶段。

- `MapXRControllerActionToRobotAction` 是一个无状态的逐帧映射，将设备输出转换为 IK 输入契约。它写入绝对基座坐标系位置，将绝对方向编码为 rotvec 目标，并将关闭度反转成电机夹爪目标：

  ```python
  action["ee.x"], action["ee.y"], action["ee.z"] = ee_pose[:3]   # 绝对位置，基座坐标系 [米]
  action["ee.wx"], action["ee.wy"], action["ee.wz"] = orient_rotvec  # 方向目标（rotvec）
  action["ee.gripper_pos"] = (1 - closedness) * 100   # 电机单位；SO-101 标定 100 = 打开
  ```

  夹爪极性（`100 = 打开, 0 = 关闭`）是源码中的硬件标定约定——如果夹爪在该关的时候打开，请在那里翻转它。

- `EEBoundsAndSafety` 将末端执行器限制在工作空间内，并对每帧跳跃进行速率限制。离合器的无瞬移特性保持帧之间变化很小，因此 `max_ee_step_m` 主要用于捕获偶发的控制器跟踪异常。Z 轴底限为 `0.0`（桌面平面），因此异常目标不会将 EE 驱动到桌面以下；x/y 保持在宽松的 `[-1, 1]` 米范围内。设置 `raise_on_jump=False` 使得超限帧被**截断并警告**而不是抛出异常——循环中间崩溃会使机械臂失去控制：

  ```python
  EEBoundsAndSafety(
      end_effector_bounds={"min": [-1.0, -1.0, 0.0], "max": [1.0, 1.0, 1.0]},
      max_ee_step_m=0.10,
      raise_on_jump=False,
  )
  ```

- `InverseKinematicsEEToJoints(initial_guess_current_joints=False, orientation_weight=0.01)` 求解闭环 Placo IK。SO-101 是一个 5 自由度机械臂，因此 IK 以位置为主；小的 `orientation_weight` 让它软性跟踪 `ee.w*` 中携带的方向目标，使手腕跟随手部，而欠约束的翻转轴则按设计部分跟踪。**没有** `GripperVelocityToJoint`：绝对的 `ee.gripper_pos` 直接传递给 `gripper.pos`。`initial_guess_current_joints=False` 使每次求解从**上一次 IK 解**热启动，而不是从测量关节值重新播种，这样关节轨迹在帧间保持连续。在硬件上调节 `orientation_weight`——太高会干扰位置跟踪，太低会忽略方向指令。

该示例还在循环级别设置了安全门控：启动复位缓动后（默认启用——传递 `--reset_to_origin=false` 保持机械臂当前位置），它**仅在离合器接合时**命令机器人，在脱离时重新发送测量关节值，因此松开离合器会使机械臂保持在原位。

参见[用于机器人和遥操作器的处理器](./processors_robots_teleop)指南，了解如何将该流程适配到其他机器人。

## 故障排除

- **`ModuleNotFoundError: isaacteleop`** —— `isaacteleop` 包未在当前环境中安装。重新运行本指南顶部的安装命令：
  `uv pip install "isaacteleop[cloudxr,retargeters-lite]~=1.3.131"`。
- **找不到控制器** —— 确保 CloudXR 运行时正在运行，防火墙端口已白名单，头戴设备已连接（参见[设置 CloudXR 并连接头戴设备](#set-up-cloudxr-and-connect-a-headset)和 Isaac Teleop [快速入门](https://nvidia.github.io/IsaacTeleop/main/getting_started/quick_start.html)）。
- **CloudXR 自动启动失败** —— 如果运行时未在其启动超时内启动，`connect()` 会抛出 `RuntimeError`。检查 `~/.cloudxr/logs` 下的启动器日志。常见原因：从未接受 EULA（以交互方式运行 `python -m isaacteleop.cloudxr --accept-eula` 一次——自动启动会在 stdin 上提示并在无头环境下挂起），或者运行时已在外部运行（设置 `LEROBOT_CLOUDXR_SKIP_AUTOLAUNCH=1` 或 `auto_launch_cloudxr=False` 以跳过自动启动）。
- **机械臂不移动** —— 离合器是一个死人手柄：你必须将握持/抓握保持在 `clutch_threshold` 以上。如果你的控制器的握持值报告偏低，请降低阈值。
- **运动感觉不对齐** —— 确认头戴设备/游戏空间的方向。控制器流通过 `XRControllerConfig` 上的 `base_T_anchor` 变换重基到机器人基座坐标系（默认：标准 OpenXR → 机器人轴约定）；如果你的锚定帧不同，请调整它。
