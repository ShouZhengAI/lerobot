# 标注流水线（Annotation Pipeline）

`lerobot-annotate` 使用视觉语言模型（VLM）观看每个 episode 的视频，并将自然语言标注写回你的数据集中。它填充[语言列与配方](./language_and_recipes)页面中介绍的两个语言列——`language_persistent` 和 `language_events`——直接写入 `data/chunk-*/file-*.parquet`。

简而言之：指向一个 LeRobot 数据集，它就会添加子任务、计划、记忆、插话、语音和视觉问答内容，供策略训练使用。

## 整体架构

```text
  你的数据集                  lerobot-annotate
  (LeRobot v3.1)
        │
        ▼
  ┌─────────────────────────────────────────────────────┐
  │                    读取 episodes                      │
  └──────────────────────────┬──────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                     ▼
  ┌──────────┐      ┌───────────────┐        ┌──────────┐       共享的 Qwen-VL
  │  plan    │      │ interjections │        │   vqa    │  ◀──  服务器（vLLM、OpenAI
  └────┬─────┘      └───────┬───────┘        └────┬─────┘       API）驱动所有三个模块
       └────────────────────┼─────────────────────┘
                            │   每个模块将原始 JSONL 暂存到
                            ▼   .annotate_staging/ 目录
                  ┌─────────────────┐
                  │    验证器        │  ◀──  检查所有内容
                  └────────┬────────┘
                           ▼
                  ┌─────────────────┐
                  │    写入器        │
                  └────────┬────────┘
                           ▼
              data/chunk-*/file-*.parquet
              (+ meta/info.json 工具)
```

三个模块（`plan`、`interjections`、`vqa`）都与**一个**共享的 VLM 通信。每个模块将其输出暂存到磁盘，验证器检查无误后，一个写入器就地重写数据集的分片（shard）。

## 流水线的输出内容

每个模块产生若干种标注（称为"样式"），路由到两个语言列之一：

| 样式 / 原子                               | 列                   | 模块             |
| ----------------------------------------- | -------------------- | ---------------- |
| `subtask`（Pi0.7 风格的"如何做，而非做什么"）| `language_persistent` | `plan`           |
| `plan`（初始 + 插话时刷新）                | `language_persistent` | `plan`           |
| `memory`（MEM 风格的压缩）                 | `language_persistent` | `plan`           |
| `task_aug`（任务描述的改写）                | `language_persistent` | `plan`           |
| `interjection`                            | `language_events`     | `interjections`  |
| 语音工具调用原子（`style=null`, `say`）    | `language_events`     | `interjections`  |
| `vqa`（用户/助手配对）                      | `language_events`     | `vqa`            |

### 子任务的生成方式

`plan` 模块并非一次性要求 VLM 生成子任务。它采用两步走的方式：**描述 → 分段**：

1. **描述（Describe）** — VLM 仅描述它在所选相机中实际看到的内容（不对任务进行猜测）。
2. **分段（Segment）** - 将该描述重新输入，VLM 将 episode 拆分为连续的原子子任务。

两个阶段都看到的是**带时间戳的缩略图（contact sheet）**——帧按 `frames_per_second`（默认 0.5 秒）采样，打包成 JPEG 网格，每帧的时间戳烧录在角落，这样 VLM 可以直接引用精确的边界时间。这比每帧一张图片在视觉 token 上的开销小得多，因此采样可以保持密集；超过 `max_frames_per_prompt` 的 episode 会以相同密度拆分为窗口并合并。两个提示都包含一个因果性的**事件边界**定义（当一个物体被拿起/放下、到达新位置、盖子状态变化或内容移动时，新事件开始），以使分割位置的判断更加精确。

可选地，第三轮**种子引导重新标注**（`--plan.subtask_seeded_relabel`）使用前一/当前/下一段的缩略图重新检查每个跨度，并对标签进行最小修正，以第一个标签为先验——它保持边界不变，仅优化措辞，代价是每个子任务多一次调用。

生成的片段随后拼接成一个无间隙、覆盖全 episode 的标注，因此**每个帧恰好有一个活跃的子任务**。参见 [`run_hf_job.py`](https://github.com/huggingface/lerobot/blob/main/examples/annotations/run_hf_job.py) 了解生产设置（单相机、带时间戳的缩略图、自动窗口化的子任务生成）。

### 工具

写入器**不会**在 parquet 中添加 `tools` 列。工具目录位于 `meta/info.json["tools"]`（参见[工具](./tools)）。每次运行后，流水线确保标准的 `say` 模式（schema）在该列表中存在，同时保留你之前声明的任何工具。

想添加自己的工具？直接编辑 `meta/info.json["tools"]`——流水线会保留已有的所有内容。这使工具对聊天模板（chat template）可见，因此模型可以学会生成该调用。实际执行生成调用的运行时层（`Tool` 协议 / `src/lerobot/tools/` 下的 `TOOL_REGISTRY`）不属于本文档范围——[工具](./tools)文档将这些部分标记为尚未实现。

## 在 Hugging Face Jobs 上运行

标注在 [Hugging Face Jobs](https://huggingface.co/docs/hub/en/jobs) 上运行。仓库提供了一个启动脚本，你可以复制并根据数据集进行调整：

```bash
HF_TOKEN=hf_... uv run python examples/annotations/run_hf_job.py
```

[`run_hf_job.py`](https://github.com/huggingface/lerobot/blob/main/examples/annotations/run_hf_job.py)
启动一个单 GPU `h200` 任务（大数据集可升级为 `h200x4`），该任务：

1. 安装 `lerobot`（来自 `main` 分支）及标注附加依赖，
2. 每个 GPU 启动一个 vLLM 服务器（使用 `vllm/vllm-openai` 镜像），通过 OpenAI 兼容 API 驱动，
3. 在数据集上使用 `lerobot-annotate` 运行 `plan` / `interjections` / `vqa` 模块，
4. 如果使用了 `--push_to_hub=true`，则将结果上传到 `--new_repo_id`（如果未指定则上传回 `--repo_id`）。

要使用不同的数据集、模型或 Hub 仓库，请编辑脚本中的 `CMD` 块。那里的每个标志直接对应一个 `lerobot-annotate` 标志（运行 `lerobot-annotate --help` 查看完整列表）。

## 关键选项

以下是你最常使用的标志。运行 `lerobot-annotate --help` 查看所有其他选项；默认值针对短操作 episodes 进行了调优。

### 数据集输入/输出

| 标志                | 默认值   | 说明                                                                 |
| ------------------- | ------- | -------------------------------------------------------------------- |
| `--repo_id`         | —       | 要标注的 Hub 数据集（如果未设置 `--root` 则下载）。                    |
| `--root`            | —       | 改为标注本地数据集目录。                                              |
| `--new_repo_id`     | —       | 将结果推送到新仓库（保留源仓库不变）。                                  |
| `--push_to_hub`     | `false` | 标注后上传（到 `--new_repo_id`，否则上传回 `--repo_id`）。              |
| `--only_episodes`   | 全部    | 仅标注这些 episode 索引（便于测试运行）。                               |
| `--seed`            | `1729`  | 为选择插话时间戳和 VQA 问题类型的随机数生成器提供种子。                  |

### 运行哪些模块

所有模块默认启用，可独立开关（设为 `false` 可跳过，例如一次只迭代一个模块）：

| 标志                        | 默认值   | 关闭的内容                           |
| --------------------------- | ------- | ----------------------------------- |
| `--plan.enabled`            | `true`  | subtasks + plan + memory + task_aug |
| `--interjections.enabled`   | `true`  | interjections + speech atoms        |
| `--vqa.enabled`             | `true`  | VQA 配对                            |

### VLM（`--vlm.*`）

| 标志                          | 默认值               | 说明                                                                               |
| --------------------------- | ------------------- | ---------------------------------------------------------------------------------- |

| `--vlm.num_gpus`            | `0`                 | 每台服务器的 GPU 数（`0` = 各一个）。                                                  |
| `--vlm.client_concurrency`  | `16`                | 所有服务器上的并发请求数。                                                            |

| 标志                                | 默认值        | 说明                                                                                                   |
| ----------------------------------- | ------------ | ------------------------------------------------------------------------------------------------------ |
| `--plan.frames_per_second`          | `2.0`        | 缩略图的帧采样率（`2.0` = 每 0.5 秒一帧）。                                                              |
| `--plan.max_frames_per_prompt`      | `60`         | 每次 VLM 调用的帧预算。采样超过此值的 episode 以相同密度自动拆分为窗口后拼接。                              |
| `--plan.contact_sheet_columns`      | `5`          | 缩略图网格的列数（每张缩略图 `contact_sheet_frames_per_sheet` 个瓦片，按行主序排列时间）。                    |

| `--plan.subtask_describe_first`     | `true`       | 运行描述→分段的基础定位流程（最佳子任务质量；+1 次调用/episode）。                                        |
| `--plan.subtask_seeded_relabel`     | `false`      | 第二轮：从每个子任务的前一/当前/下一段缩略图重新标注，以第一轮标签为种子（+1 次调用/子任务）。                |

| `--plan.derive_task_from_video`     | `if_short`   | 按原样使用数据集任务（`off`）、仅在任务缺失/过短时重新推导（`if_short`）、或始终从视频重新推导（`always`）。 |

## 贡献新模块

该流水线设计为可扩展的，**非常欢迎贡献**——无论是全新的模块（例如轨迹跟踪或 affordance）、新的提示模板、更智能的基础定位流程，还是对现有 `plan` / `interjections` / `vqa` 模块的质量修复。

每个模块位于 `src/lerobot/annotations/steerable_pipeline/modules/` 下，共享 VLM 客户端和关键帧缓存，将原始输出写入暂存树，并作为自己的阶段插入执行器。有想法？请在[仓库](https://github.com/huggingface/lerobot)上提交 issue 或 PR。

## 配方（recipe）如何使用输出

标注数据供配方使用（参见[语言列与配方](./language_and_recipes)）。通常：

- 低级/高级/记忆更新分支从 `language_persistent` 读取 `subtask` / `plan` / `memory`。
- 插话响应分支读取 `interjection` 事件及其配对的语音原子（通过 `tool_calls_from` 合并为一个助手轮次）以及在相同时间戳对应的 `plan` 刷新。
- VQA 分支从 `language_events` 读取 `(vqa, user)` 和 `(vqa, assistant)` 配对。

## 为什么将状态和事件分开

设计基于两个理念：

1. **持久状态 vs. 精确事件。** 持久行（`subtask`、`plan`、`memory`）适用于整个 episode，回答"现在的情况是什么？"。事件行（`interjection`、`vqa`、speech）仅出现在时间戳匹配的帧上。时间戳直接从源 parquet 复制——绝不通过浮点数重新计算。
2. **一次 VLM 推理。** 所有三个模块共享一个 VLM 客户端（与任务 vLLM 服务器通信的 OpenAI 兼容客户端），因此你为每个数据集只支付一次模型加载成本，而不是三次。

## 重新运行单个模块

每个模块将其原始输出暂存到 `<root>/.annotate_staging/episode_{N:06d}/<module>.jsonl`。这使得提示迭代非常高效：重新运行一个模块只会覆盖其自身的 JSONL，然后写入器重新组合最终的 parquet。使用 `--plan.enabled=false`（以及相应的 `--interjections.enabled` / `--vqa.enabled`）禁用在测试时不需要的模块，一次只测试一个。

## 验证器的检查内容

在写入器运行之前，`StagingValidator` 确认：

- 每个事件行恰好落在真实帧时间戳上；
- 没有语音/插话配对被孤立；
- 在每个插话时间戳上刷新了 `plan`；
- `memory` 行落在子任务边界上（警告，而非错误）；
- 每个 VQA 助手的 `content` 是有效的 JSON，属于 bbox / keypoint / count / attribute / spatial 形状之一；
- 每行都转到 `column_for_style(style)` 选择的列。

任何错误都会中止写入器。在调试时传递 `--skip_validation=true` 以覆盖这些检查。

## 每个模块的思想来源

- **`plan` — subtasks。** Hi Robot（[Shi 2025](https://arxiv.org/abs/2502.19417)）提供了原子粒度（"拿起一片生菜"、"将碗放入盒子"）；Pi0.7（[Physical Intelligence 2025](https://pi.website/pi07)）提供了"如何做，而非做什么"的细节。
- **`plan` — memory。** MEM（[Torne 2026](https://arxiv.org/abs/2603.03596)）：仅保留最小相关信息——保留结果，丢弃具体属性。
- **`interjections`。** Hi Robot 的场景分类：负面任务、情境修正、特定约束、偏好。语音是仅含工具调用的原子（`tool_calls=[{type:function, function:{name:"say", arguments:{text:...}}}]`）。
- **`vqa`。** ECoT（[Zawalski 2024](https://arxiv.org/abs/2407.08693)）提供了基础定位特征（像素边界框 `[x_min, y_min, x_max, y_max]`、关键点），Steerable VLA Policies（[Zhao 2025](https://arxiv.org/abs/2509.07626)）提供了多抽象层次的基础定位。Pi0.7 也跨抽象层次对答案进行基础定位。

改进模块时，建议调整其提示模板（位于 `src/lerobot/annotations/steerable_pipeline/prompts/`），而不是从头重写。

## 大致成本估算

每个 episode，流水线进行约 `max_steps` 次计划调用、`max_interjections_per_episode` 次插话调用和 `vqa_emission_hz × episode_seconds` 次 VQA 调用。以默认设置（8 个子任务、1 次插话、1 Hz × 3 对）在 30 秒的 episode 上，大约是 50 次 VLM 调用。

存储占用很小：`language_persistent` 每个 episode 最多几十 KB（parquet 对跨帧重复的一条条目进行字典编码），而 `language_events` 在大多数帧上为空——其大小随输出次数扩展，而非 `num_frames × num_emissions`。
