# 调试处理管线（Processor Pipeline）

处理管线可能非常复杂，尤其是在链式组合多个转换步骤时。与简单的函数调用不同，管线缺乏天然的可观测性——你无法轻易看到每个步骤之间的处理过程，也难以定位问题所在。本指南提供了专门设计的调试工具和技术，帮助你应对这些挑战，理解数据在管线中的流动过程。

我们将探讨三种互补的调试方法：用于运行时监控的**钩子（Hook）**、用于详细检查的**逐步调试（Step-through Debugging）**，以及用于捕获结构不匹配问题的**特征验证（Feature Validation）**。每种方法服务于不同的目的，三者结合可以为你的管线行为提供全方位的可见性。

## 理解钩子（Hook）

钩子是在管线执行过程中特定时刻被调用的函数。它们提供了一种在不修改管线代码的情况下检查、监控或修改数据的方式。可以将其视为管线的"事件监听器"。

### 什么是钩子（Hook）？

钩子是一个回调函数，在管线执行的特定时刻被自动调用。这个概念源于事件驱动编程——你可以想象自己"钩入"管线的执行流来观察或响应正在发生的事情。

可以把钩子想象成在管线中插入的检查点。每当管线到达其中一个检查点时，它会短暂暂停并调用你的钩子函数，让你有机会检查当前状态、记录信息并验证数据。

钩子就是一个接受两个参数的函数：

- `step_idx: int` — 当前处理步骤的索引（0, 1, 2 等）
- `transition: EnvTransition` — 管线在该点的数据转换（transition）

钩子之美在于其非侵入性：你可以在不修改管线代码的情况下添加监控、验证或调试逻辑。管线保持干净并专注于核心逻辑，而钩子负责处理日志记录、监控和调试等横切关注点。

### 前置钩子与后置钩子

管线支持两种类型的钩子：

- **前置钩子**（`register_before_step_hook`）— 在每个步骤执行前调用
- **后置钩子**（`register_after_step_hook`）— 在每个步骤完成后调用

```python
def before_hook(step_idx: int, transition: EnvTransition):
    """在步骤处理 transition 之前调用。"""
    print(f"即将执行步骤 {step_idx}")
    # 用途：日志记录、验证、设置

def after_hook(step_idx: int, transition: EnvTransition):
    """在步骤处理完 transition 之后调用。"""
    print(f"步骤 {step_idx} 已完成")
    # 用途：结果监控、清理、调试

processor.register_before_step_hook(before_hook)
processor.register_after_step_hook(after_hook)
```

### 实现 NaN 检测钩子

以下是一个检测 NaN 值的钩子示例：

```python
def check_nans(step_idx: int, transition: EnvTransition):
    """检查观测值中是否存在 NaN 值。"""
    obs = transition.get(TransitionKey.OBSERVATION)
    if obs:
        for key, value in obs.items():
            if isinstance(value, torch.Tensor) and torch.isnan(value).any():
                print(f"在步骤 {step_idx} 的 {key} 中检测到 NaN")

# 注册钩子，使其在每一步之后执行
processor.register_after_step_hook(check_nans)

# 处理数据——钩子会自动被调用
output = processor(input_data)

# 调试完成后移除钩子
processor.unregister_after_step_hook(check_nans)
```

### 钩子的内部工作原理

了解内部机制有助于你更有效地使用钩子。管线维护了两个独立的列表：一个用于前置钩子，另一个用于后置钩子。当你注册一个钩子时，它会被追加到相应的列表中。

在执行过程中，管线遵循严格的顺序：对于每个处理步骤，它首先按注册顺序调用所有前置钩子，然后执行实际的步骤转换，最后按注册顺序调用所有后置钩子。这就在每个步骤周围形成了一个可预测的"三明治"式结构。

关键在于钩子不会改变管线的核心逻辑——它们是纯粹的附加功能。管线的 `_forward` 方法协调钩子与处理步骤之间的配合，确保你的调试或监控代码在完全正确的时刻运行，而不会干扰主数据流。

以下是管线执行钩子的简化示意：

```python
class DataProcessorPipeline:
    def __init__(self):
        self.steps = [...]
        self.before_step_hooks = []  # 前置钩子列表
        self.after_step_hooks = []   # 后置钩子列表

    def _forward(self, transition):
        """在所有步骤中处理 transition 的内部方法。"""
        for step_idx, processor_step in enumerate(self.steps):
            # 1. 调用所有前置钩子
            for hook in self.before_step_hooks:
                hook(step_idx, transition)

            # 2. 执行实际的处理步骤
            transition = processor_step(transition)

            # 3. 调用所有后置钩子
            for hook in self.after_step_hooks:
                hook(step_idx, transition)

        return transition

    def register_before_step_hook(self, hook_fn):
        self.before_step_hooks.append(hook_fn)

    def register_after_step_hook(self, hook_fn):
        self.after_step_hooks.append(hook_fn)
```

### 执行流程

执行流程如下所示：

```
输入 → 前置钩子 → 步骤 0 → 后置钩子 → 前置钩子 → 步骤 1 → 后置钩子 → ... → 输出
```

例如，包含 3 个步骤和两种钩子的情况：

```python
def timing_before(step_idx, transition):
    print(f"⏱️  开始步骤 {step_idx}")

def validation_after(step_idx, transition):
    print(f"✅ 完成步骤 {step_idx}")

processor.register_before_step_hook(timing_before)
processor.register_after_step_hook(validation_after)

# 输出：
# ⏱️  开始步骤 0
# ✅ 完成步骤 0
# ⏱️  开始步骤 1
# ✅ 完成步骤 1
# ⏱️  开始步骤 2
# ✅ 完成步骤 2
```

### 多钩子

你可以注册多个相同类型的钩子——它们按注册顺序执行：

```python
def log_shapes(step_idx: int, transition: EnvTransition):
    obs = transition.get(TransitionKey.OBSERVATION)
    if obs:
        print(f"步骤 {step_idx} 观测值形状：")
        for key, value in obs.items():
            if isinstance(value, torch.Tensor):
                print(f"  {key}: {value.shape}")

processor.register_after_step_hook(check_nans)      # 第一个执行
processor.register_after_step_hook(log_shapes)     # 第二个执行

# 两个钩子会在每一步之后按注册顺序被调用
output = processor(input_data)
```

虽然钩子非常适合监控特定问题（如 NaN 检测）或在管线正常运行期间收集指标，但有时你需要更深入的分析。当你想了解每个步骤的详细处理过程或调试复杂的转换逻辑时，逐步调试提供了所需的详细检查能力。

## 逐步调试

逐步调试好比为管线提供了慢动作回放功能。你不需要眼睁睁看着数据从输入到输出瞬间完成转换，而是可以在每个步骤之后暂停并检查处理结果。

这种方法在你试图理解复杂管线、调试意外行为或验证每个转换是否符合预期时尤其有用。与适合自动化监控的钩子不同，逐步调试为你提供了手动、交互式的检查过程控制权。

`step_through()` 方法是一个生成器（generator），它在每个处理步骤之后产生 transition 状态，让你可以检查中间结果。可以将它想象成在数据流经管线时创建一系列数据快照——每个快照都精确展示了你的数据在又经过一次转换后的样子。

### 逐步调试的工作原理

`step_through()` 方法从根本上改变了管线的执行方式。它不是按顺序运行所有步骤并只返回最终结果，而是将管线转换为一个迭代器，逐个产生中间结果。

内部执行过程如下：该方法首先将输入数据转换为管线内部的 transition 格式，然后产出这个初始状态。接下来，它应用第一个处理步骤并产出结果。然后对结果应用第二个步骤并再次产出，依此类推。每次 `yield` 都会给出该时刻 transition 的完整快照。

这种生成器模式之所以强大，是因为它是惰性的——管线只在你请求时才计算下一步。这意味着你可以随时停止，彻底检查当前状态，并决定是否继续。你无需为了调试一个有问题的步骤而运行整个管线。

`step_through()` 不会运行整个管线并只展示最终结果，而是在每一步之后暂停并给出中间的 transition：

```python
# 这会创建一个产出中间状态的生成器
for i, intermediate_result in enumerate(processor.step_through(input_data)):
    print(f"=== 步骤 {i} 之后 ===")

    # 检查该阶段的观测值
    obs = intermediate_result.get(TransitionKey.OBSERVATION)
    if obs:
        for key, value in obs.items():
            if isinstance(value, torch.Tensor):
                print(f"{key}: shape={value.shape}, dtype={value.dtype}")
```

### 使用断点进行交互式调试

你可以在逐步调试循环中添加断点来进行交互式调试：

```python
# 逐步调试管线
for i, intermediate in enumerate(processor.step_through(data)):
    print(f"步骤 {i}: {processor.steps[i].__class__.__name__}")

    # 设置断点以检查当前状态
    breakpoint()  # 调试器将在此处暂停

    # 你现在可以在调试器中检查 'intermediate'：
    # - 检查张量形状和值
    # - 验证预期转换
    # - 查找意外变化
```

在调试器会话中，你可以：

- 检查 `intermediate[TransitionKey.OBSERVATION]` 来查看观测值数据
- 检查 `intermediate[TransitionKey.ACTION]` 来查看动作变换
- 检查 transition 的任何部分以了解每个步骤的作用

逐步调试非常适合理解数据的_值_变换，但数据的_结构_呢？虽然钩子和逐步调试可以帮助你调试运行时行为，但你还需确保管线生成的数据格式与下游组件兼容。这就是特征契约验证发挥作用的地方。

## 验证特征契约

特征契约定义了管线期望输入什么数据结构，以及输出什么数据结构。验证这些契约有助于及早捕获不匹配问题。

### 理解特征契约

每个处理步骤都有一个 `transform_features()` 方法，描述它如何改变数据结构：

```python
# 获取管线的预期输出特征
initial_features = {
    PipelineFeatureType.OBSERVATION: {
        "observation.state": PolicyFeature(type=FeatureType.STATE, shape=(7,)),
        "observation.image": PolicyFeature(type=FeatureType.IMAGE, shape=(3, 224, 224))
    },
    PipelineFeatureType.ACTION: {
        "action": PolicyFeature(type=FeatureType.ACTION, shape=(4,))
    }
}

# 检查管线的输出结果
output_features = processor.transform_features(initial_features)

print("输入特征：")
for feature_type, features in initial_features.items():
    print(f"  {feature_type}:")
    for key, feature in features.items():
        print(f"    {key}: {feature.type.value}, shape={feature.shape}")

print("\n输出特征：")
for feature_type, features in output_features.items():
    print(f"  {feature_type}:")
    for key, feature in features.items():
        print(f"    {key}: {feature.type.value}, shape={feature.shape}")
```

### 验证预期特征

检查你的管线是否生成了你期望的特征：

```python
# 定义你期望管线生成的特征
expected_keys = ["observation.state", "observation.image", "action"]

print("正在验证特征契约...")
for expected_key in expected_keys:
    found = False
    for feature_type, features in output_features.items():
        if expected_key in features:
            feature = features[expected_key]
            print(f"✅ {expected_key}: {feature.type.value}, shape={feature.shape}")
            found = True
            break

    if not found:
        print(f"❌ 缺少预期特征: {expected_key}")
```

这种验证有助于确保你的管线能够与期望特定数据结构的其他组件正确配合。

## 总结

现在你已经了解了三种调试方法，可以系统性地解决任何管线问题：

1. **钩子（Hook）** — 无需修改管线代码即可进行运行时监控和验证
2. **逐步调试（Step-through Debugging）** — 检查中间状态并理解转换过程
3. **特征验证（Feature Validation）** — 确保满足数据结构契约

**何时使用每种方法：**

- 需要了解管线做了什么或出现意外情况时，从**逐步调试**开始
- 在开发和部署过程中添加**钩子**进行持续监控，自动捕获问题
- 在部署前使用**特征验证**确保管线与下游组件兼容

这三种工具协同工作，为复杂管线提供天然缺失的完整可观测性。用钩子监控问题，用逐步调试理解行为，用特征验证确保兼容性——你就能自信且高效地调试任何管线。
