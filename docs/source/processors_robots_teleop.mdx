# 用于机器人和遥操作的数据处理器

本指南介绍如何构建和修改将遥操作设备（例如手机）连接到机器人和数据集的数据处理管线。这些管线标准化了不同动作/观测空间之间的转换，使您可以更换遥操作设备和机器人而无需重写胶水代码。

我们以手机到 SO‑100 从动臂的示例进行具体说明，但同样的模式也适用于其他机器人。

**您将学到什么**

- **绝对与相对末端执行器控制**：各自的含义、权衡以及如何为您的任务做出选择。
- **三管线模式**：如何将遥操作动作映射为数据集动作，再映射为机器人指令，以及将机器人观测映射为数据集观测。
- **适配器（`to_transition` / `to_output`）**：如何将原始字典转换为 `EnvTransition` 并返回，以减少样板代码。
- **数据集特征合约**：步骤如何通过 `transform_features(...)` 声明特征，以及如何为录制聚合和合并特征。
- **选择表示方式**：何时存储关节位置、绝对末端执行器姿态或相对末端执行器增量，以及这对训练的影响。
- **管线定制指南**：如何安全更换机器人/URDF，以及调整边界、步长和逆运动学初始化等选项。

### 绝对与相对末端执行器控制

本指南中的示例使用绝对末端执行器（end effector, EE）姿态，因为它们易于理解。在实际应用中，相对末端执行器增量或关节位置通常是更受青睐的学习特征。

使用处理器，您可以为策略选择所需的学习特征。可以是关节位置/速度、绝对末端执行器姿态或相对末端执行器位置。您也可以选择存储其他特征，例如关节力矩、电机电流等。

## 三条管线

我们通常会组合三条管线。根据您的具体设置，如果动作和观测空间已经匹配，其中一些管线可以为空。
每条管线处理不同动作和观测空间之间的转换。以下是每条管线的简要说明。

1. **管线 1**：遥操作动作空间 → 数据集动作空间（手机姿态 → 末端执行器目标）
2. **管线 2**：数据集动作空间 → 机器人指令空间（末端执行器目标 → 关节）
3. **管线 3**：机器人观测空间 → 数据集观测空间（关节 → 末端执行器姿态）

以下是在手机到 SO-100 从动臂示例中使用的三条管线示例：

```python
phone_to_robot_ee_pose_processor = RobotProcessorPipeline[RobotAction, RobotAction]( # 遥操作 -> 数据集动作
    steps=[
        MapPhoneActionToRobotAction(platform=teleop_config.phone_os),
        EEReferenceAndDelta(
            kinematics=kinematics_solver, end_effector_step_sizes={"x": 0.5, "y": 0.5, "z": 0.5}, motor_names=list(robot.bus.motors.keys()),
        ),
        EEBoundsAndSafety(
            end_effector_bounds={"min": [-1.0, -1.0, -1.0], "max": [1.0, 1.0, 1.0]}, max_ee_step_m=0.20,
        ),
        GripperVelocityToJoint(),
    ],
    to_transition=robot_action_to_transition,
    to_output=transition_to_robot_action,
)

robot_ee_to_joints_processor = RobotProcessorPipeline[RobotAction, RobotAction]( # 数据集动作 -> 机器人
    steps=[
        InverseKinematicsEEToJoints(
            kinematics=kinematics_solver, motor_names=list(robot.bus.motors.keys()), initial_guess_current_joints=True,
        ),
    ],
    to_transition=robot_action_to_transition,
    to_output=transition_to_robot_action,
)

robot_joints_to_ee_pose = RobotProcessorPipeline[RobotObservation, RobotObservation]( # 机器人观测 -> 数据集观测
    steps=[
        ForwardKinematicsJointsToEE(kinematics=kinematics_solver, motor_names=list(robot.bus.motors.keys()))
    ],
    to_transition=observation_to_transition,
    to_output=transition_to_observation,
)
```

## 为什么使用 to_transition / to_output

为了在机器人/遥操作设备与管线之间进行转换，我们使用 `to_transition` 和 `to_output` 管线适配器。
它们标准化了转换过程以减少样板代码，并充当了机器人和遥操作设备原始字典与管线 `EnvTransition` 格式之间的桥梁。
在手机到 SO-100 从动臂的示例中，我们使用以下适配器：

- `robot_action_to_transition`：将遥操作动作字典转换为管线的 transition。
- `transition_to_robot_action`：将管线的 transition 转换为机器人动作字典。
- `observation_to_transition`：将机器人观测字典转换为管线的 transition。
- `transition_to_observation`：将管线的 transition 转换为观测字典。

更多详情请查看 [src/lerobot/processor/converters.py](https://github.com/huggingface/lerobot/blob/main/src/lerobot/processor/converters.py)。

## 数据集特征合约

数据集特征由存储在数据集中的键决定。每个步骤都可以在名为 `transform_features(...)` 的合约中声明它修改哪些特征。构建好处理器后，处理器可以通过 `aggregate_pipeline_dataset_features()` 聚合所有这些特征，并通过 `combine_feature_dicts(...)` 合并多个特征字典。

以下是在手机到 SO-100 从动臂示例中，我们如何使用 `transform_features` 方法声明特征的示例：

```python
    def transform_features(
        self, features: dict[PipelineFeatureType, dict[str, PolicyFeature]]
    ) -> dict[PipelineFeatureType, dict[str, PolicyFeature]]:
        # 我们只在数据集中使用末端执行器姿态，因此不需要关节位置
        for n in self.motor_names:
            features[PipelineFeatureType.ACTION].pop(f"{n}.pos", None)
        # 指定我们希望在此步骤中存储在数据集中的数据集特征
        for k in ["x", "y", "z", "wx", "wy", "wz", "gripper_pos"]:
            features[PipelineFeatureType.ACTION][f"ee.{k}"] = PolicyFeature(
                type=FeatureType.STATE, shape=(1,)
            )
        return features
```

这里我们声明了在此步骤中修改了哪些 PolicyFeature，这样我们就可以预期运行处理器时会得到哪些特征。这些特征随后可以被聚合，并用于创建数据集特征。

以下是在手机到 SO-100 录制示例中，我们如何聚合和合并特征的示例：

```python
features=combine_feature_dicts(
        # 运行管线的特征合约
        # 这告诉您经过管线步骤后特征会是什么样子
        aggregate_pipeline_dataset_features(
            pipeline=phone_to_robot_ee_pose_processor,
            initial_features=create_initial_features(action=phone.action_features), # <- 我们可以预期的动作特征，来自我们的遥操作设备（手机）和动作处理器
            use_videos=True,
        ),
        aggregate_pipeline_dataset_features(
            pipeline=robot_joints_to_ee_pose,
            initial_features=create_initial_features(observation=robot.observation_features), # <- 我们可以预期的观测特征，来自我们的机器人和观测处理器
            use_videos=True,
            patterns=["observation.state.ee"], # <- 在此处您可以选择性地使用特定模式过滤要存储在数据集中的特征

        ),
    ),
```

工作原理如下：

- `aggregate_pipeline_dataset_features(...)`：在管线上应用 `transform_features`，并按模式过滤（当 `use_videos=True` 时包含图像，当指定了 `patterns` 时包含状态特征）。
- `combine_feature_dicts(...)`：合并多个特征字典。
- 使用 `record_loop(...)` 录制时，会先用 `build_dataset_frame(...)` 构建与 `dataset.features` 一致的帧，然后再调用 `add_frame(...)` 将该帧添加到数据集中。

## 自定义机器人管线的指南

您可以选择以下任何一种特征作为您的动作/观测空间：

- 关节位置
- 绝对末端执行器姿态
- 相对末端执行器增量
- 其他特征：关节速度、力矩等

根据您为策略动作和观测空间选择的内容，配置和修改相应的管线与步骤。

### 不同机器人

- 您可以轻松复用管线，例如将另一台机器人用于手机遥操作：修改示例，更换机器人 `RobotKinematics`（URDF）和 `motor_names`，即可将您自己的机器人用于手机遥操作。此外，请确保 `target_frame_name` 指向您的夹爪/腕部。

### 安全第一

- 更改管线时，从较紧的边界开始，并在使用真实机器人时实现安全步骤。
- 建议先从仿真开始，然后再迁移到真实机器人。
