# LingBot-VA

LingBot-VA 是一个基于 **Wan2.2** 视频扩散框架构建的**自回归视频-动作世界模型策略**。它在一个自回归序列中交错预测未来的**视频潜变量**和**机器人动作**（"VA" = Video-Action，视频-动作）。LeRobot 集成将 LingBot-VA 接入标准的训练、评估和处理器接口。

## 模型概述

LingBot-VA 是一个**双流"混合变换器（mixture-of-transformers）"**：一个视频/潜变量流（`patch_embedding_mlp → blocks → proj_out`）和一个动作流（`action_embedder → blocks → action_proj_out`）共享相同的 30 个 Transformer 块和文本条件输入。

| 组件                    | 类                        | 作用                                               |
| ----------------------- | ------------------------- | -------------------------------------------------- |
| DiT 主干（可训练）      | `WanTransformer3DModel`   | 约 5B 参数的双流变换器。                           |
| VAE（冻结）             | `AutoencoderKLWan`        | Wan2.2 VAE，`z_dim=48`。从源仓库惰性拉取。         |
| 文本编码器（冻结）      | `UMT5EncoderModel`        | UMT5-XXL，`d_model=4096`。从源仓库惰性拉取。       |

在推理时，策略每个数据块（chunk）执行一次自回归循环：它使用两个独立的流匹配调度器对视频潜变量流（CFG，约 20 步）和动作流（约 50 步）进行去噪，并在数据块之间维护 KV 缓存。在执行数据块时，实际观测到的关键帧被反馈回 KV 缓存（闭环世界模型）。

### LeRobot 集成覆盖的内容

- 通过 LeRobot 标准配置 `policy.type=lingbot_va`。
- Hub 上现成的 LeRobot 格式检查点（从上游发布的检查点转换而来）。
- 标准 `select_action` 接口背后的自回归双流推理（单环境评估，`--eval.batch_size=1`）。
- 在评估/训练期间，可选择保存策略的**预测（想象）视频**。
- 在 LIBERO 和 RoboTwin 上使用 `lerobot-eval` 进行评估。
- 通过双流流匹配损失（`policy.forward`）进行训练/微调，详见下文。

## 安装

1. 按照[安装指南](./installation)安装 LeRobot。
2. 安装 LingBot-VA 额外组件：

```bash
pip install -e ".[lingbot_va]"
```

## 检查点

已发布的上游检查点已转换为 LeRobot 格式并推送到 Hub：

| 变体                     | LeRobot 检查点                    |
| ------------------------ | --------------------------------- |
| LIBERO-Long 后训练       | `lerobot/lingbot_va_libero_long`  |
| RoboTwin 后训练          | `lerobot/lingbot_va_robotwin`     |
| 预训练基础模型           | `lerobot/lingbot_va_base`         |

LeRobot 的 `model.safetensors` 中仅存储了可训练的约 5B 参数变换器。冻结的 VAE + UMT5 + 分词器（约 20 GB）在加载时从 `config.wan_pretrained_path` 拉取（默认为源仓库 `robbyant/*`）。UMT5-XXL 文本编码器默认在 CPU 上运行（`config.text_encoder_device`），因此 5B 参数变换器 + VAE 可放入单张 24–32 GB GPU。

## 评估（LIBERO）

```bash
lerobot-eval \
    --policy.path=lerobot/lingbot_va_libero_long \
    --policy.device=cuda \
    --env.type=libero --env.task=libero_10 \
    --env.observation_height=128 --env.observation_width=128 \
    --eval.n_episodes=50 --eval.batch_size=1 \
    --output_dir=outputs/eval/lingbot_va_libero
```

LingBot-VA 的流式推理（KV 缓存 + 观测关键帧反馈）针对单环境评估实现；请使用 `--eval.batch_size=1`。

## 评估（RoboTwin）

RoboTwin 2.0 需要 SAPIEN + CuRobo 仿真器栈。你可以使用基准测试 Docker 镜像（`docker/Dockerfile.benchmark.robotwin`，还需要 `warp-lang==1.3.1` 以及在 `TORCH_CUDA_ARCH_LIST` 中使用 GPU 的计算能力构建的 CuRobo）。RoboTwin 使用**末端执行器位姿（end-effector-pose）控制**，因此请使用 `--env.action_mode=ee` 运行：策略预测每个机械臂的 `xyz+四元数+夹爪` 增量（`robotwin_tshape` 潜变量布局），组合到片段的初始末端执行器位姿上，并通过 CuRobo 逆运动学（IK）执行。

```bash
lerobot-eval \
    --policy.path=lerobot/lingbot_va_robotwin \
    --policy.device=cuda \
    --env.type=robotwin --env.task=beat_block_hammer --env.action_mode=ee \
    --eval.n_episodes=10 --eval.batch_size=1 \
    --output_dir=outputs/eval/lingbot_va_robotwin
```

### 保存预测（想象）视频

设置 `--policy.save_predicted_video=true` 可额外对预测的视频潜变量进行 VAE 解码，并写入 `pred_episode_*.mp4` 文件，与环境渲染的 `eval_episode_*.mp4` 视频并存。该标志同样适用于 `lerobot-train` 期间的周期性评估。

## 训练/微调

`LingBotVAPolicy.forward(batch)` 实现了论文中的双流**流匹配（flow-matching）**损失（`latent_loss + action_loss`，按时间步加权，动作掩码）：它将摄像头片段 VAE 编码为视频潜变量，将任务 UMT5 编码，对两个流添加噪声，运行变换器的块因果训练前向传播，并返回 `(loss, metrics)`。优化器预设为 AdamW，使用线性预热后恒定学习率调度（与上游一致）。

要求：

- 块因果掩码使用 PyTorch **flex-attention**，因此训练时需要以 `--policy.attn_mode=flex` 构建策略（默认的 `torch` SDPA 仅用于推理）。
- 完整的 5B 参数 DiT 在 AdamW 下无法放入单张 24–32 GB GPU；请使用 **LoRA**（`--policy.use_peft=true`）和/或优化器卸载进行微调。`get_optim_params` 仅返回可训练（例如适配器）参数；VAE + UMT5 文本编码器保持冻结。

```bash
lerobot-train \
  --policy.path=lerobot/lingbot_va_libero_long --policy.attn_mode=flex \
  --policy.use_peft=true \
  --dataset.repo_id=<你的 LeRobot 格式数据集> \
  --batch_size=1 --steps=... --output_dir=outputs/train/lingbot_va
```

数据集必须提供摄像头片段（每个摄像头的一个时间窗口，VAE 编码为 `frame_chunk_size` 个潜变量帧）以及每个样本的 `frame_chunk_size * action_per_frame` 个动作步骤。

## 数据格式（动作通道与摄像头顺序）

LingBot-VA 是一个**末端执行器（笛卡尔空间）位姿**策略，它预测的是末端执行器位姿 + 夹爪，而非关节位置。动作位于固定的多形态**30 维**布局中；将你机器人的动作维度映射到这些通道中，其余部分用 `0` 填充（`used_action_channel_ids` 选择给定检查点实际使用的通道）：

| 通道    | 含义                                                  |
| ------- | ----------------------------------------------------- |
| 0–6     | 左臂末端执行器位姿                                    |
| 7–13    | 右臂末端执行器位姿                                    |
| 14–20   | 左臂关节（已发布检查点未使用）                        |
| 21–27   | 右臂关节（已发布检查点未使用）                        |
| 28      | 左夹爪                                                |
| 29      | 右夹爪                                                |

- **LIBERO** 使用通道 `0–6`：6 自由度末端执行器增量（xyz + 旋转）+ 夹爪（单臂）。
- **RoboTwin** 使用通道 `[0–6, 28, 7–13, 29]`：左端执行器（xyz + 四元数）+ 左夹爪 + 右端执行器 + 右夹爪（16 维）。环境通过 CuRobo IK 将这些位姿转换为关节轨迹——关节从不被预测。

在微调这些检查点之前，关节空间数据集（或不同的末端执行器约定）必须重新映射到此模式。

**摄像头顺序是固定的且对顺序敏感**，每个摄像头的潜变量按照 `obs_cam_keys` 顺序在空间上拼接，因此物理摄像头到槽位的映射必须与训练时一致：

| 基准测试 | `obs_cam_keys`（按顺序）                                                                     | `camera_layout`                                                     |
| -------- | -------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| LIBERO   | `observation.images.image`（agentview / 第三人称），`observation.images.image2`（腕部内置）   | `width_concat`（潜变量在宽度上拼接）                                 |
| RoboTwin | `observation.images.head_camera`，`observation.images.left_camera`，`observation.images.right_camera` | `robotwin_tshape`（全分辨率头部在下方，两个半分辨率腕部在上方）      |

第一个摄像头是外部/头部视图，其余是腕部视图。

## 推理超参数（LIBERO）

| 键                                      | 值                                                                                |
| --------------------------------------- | --------------------------------------------------------------------------------- |
| height × width                          | 128 × 128                                                                         |
| 摄像头                                  | `observation.images.image`（agentview），`observation.images.image2`（腕部内置）    |
| 使用的动作通道                          | 0–6（7 自由度机械臂 + 夹爪）                                                      |
| action_per_frame / frame_chunk_size     | 4 / 4                                                                             |
| attn_window                             | 30                                                                                |
| 视频/动作去噪步数                       | 20 / 50                                                                           |
| guidance_scale / action_guidance_scale  | 5 / 1                                                                             |
| snr_shift / action_snr_shift            | 5.0 / 0.05                                                                        |

这些是 `LingBotVAConfig` 的默认值；可通过 `--policy.<name>=...` 覆盖任意参数。

## 备注

- **注意力后端：** 推理使用 `torch` SDPA 后端（始终可用）。`flashattn` 和 `flex` 后端是可选的；`flex` 仅训练时需要。
- **模型大小：** DiT 约 5B 参数，冻结的 VAE+UMT5 约 20 GB；推理需要大约 18–24 GB 显存。
